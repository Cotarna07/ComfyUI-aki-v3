# 漫画分镜切割、排序与角色一致性方案梳理

## 这份文档解决什么

这份文档面向当前新增需求：把整页漫画自动切成单独分镜，按故事连续性排序，理解每个分镜内容，并在跨分镜范围内识别同一角色，为后续连续故事生成或角色一致性生成做准备。

结论先说：目前没有一套现成的 ComfyUI 工作流能把这四件事一次性全做完，但已经能找到足够多的现成节点，把它们拼成一条可落地的链路。最稳妥的做法不是硬找“一键全链路模板”，而是采用“分镜切分 + 快速视觉理解 + 强推理重排 + 角色注册表”的模块化方案。

## 总体判断

1. 分镜切割：有现成 ComfyUI 节点和工作流思路，能做。
2. 分镜排序：纯几何排序只能得到阅读顺序，做不到真正的故事连续性，必须加 VLM 或 LLM 二次重排。
3. 分镜内容识别：现成 Florence-2 / Qwen-VL 节点足够完成，能做结构化输出。
4. 跨图角色一致性：分成“识别是不是同一角色”和“后续生成时保持同脸”两部分；前者能做但需要组合特征，后者现成资源很多。

## 推荐总链路

建议把整条链路拆成 7 个阶段：

1. 整页漫画预处理
2. 分镜边界检测与切割
3. 生成每个分镜的 crop、polygon、bbox、候选阅读顺序
4. 对单个分镜做首轮视觉理解和文本抽取
5. 基于上下文做二轮故事连续性重排与故事摘要
6. 建立角色注册表并执行跨分镜角色归并
7. 导出结构化结果，供后续生成或检索流程使用

建议输出不要只留图片，至少保留一份 JSON 或 JSONL。后续只要没有修改原图，就不必重新跑切分、描述和角色聚类。

## 1. 分镜切割与排序

### 推荐资源 1：ComfyUI Panels

资源：

- https://github.com/bmad4ever/comfyui_panels

这个仓库不是泛泛的图像分割插件，而是专门围绕 comics / manga panels 设计的节点集合。它有几个点非常关键：

1. 支持 Panel Layout 的层级表达，不只是切出矩形框。
2. README 明确提到可以从图像中提取 panels，节点名包括 Detect Panels In Image、Build Layout Panels、Panel to Mask、Polygon To Resized Mask。
3. README 明确描述了 panel order，且支持 left-to-right 和 right-to-left 两种阅读方向。
4. 仓库里自带 sample_layouts 和 workflows，适合直接拿来做模板起步。

它最适合解决的问题是：

- 把漫画页切成可单独处理的分镜
- 给每个分镜保留几何形状和相对顺序
- 对规则版式、黑白稿、边界清晰的彩页效果更好

它的边界也很明确：

- 对“剧情连续性”没有真正理解，只有版式和切割顺序
- 对复杂跨页、出血、重叠对白、故意打乱阅读方向的页面，需要后续推理修正

### 推荐资源 2：MangaPanelSegmentationNode

资源：

- https://comfyai.run/documentation/MangaPanelSegmentationNode

这个页面描述了一个名为 MangaPanelSegmentationNode 的节点，输入包括：

- 漫画页图像
- segmentation type
- sensitivity

输出包括：

- 分镜图片列表
- segmentation map JSON

它的说明和需求方向是对的，但目前查到的是节点文档页，不是明确的官方仓库主页。因此它可以当参考模板和能力说明，但不建议把它当唯一依赖。更稳妥的做法仍然是优先以 comfyui_panels 为主。

### 切割阶段的工程建议

为了兼容黑白漫画和彩色漫画，建议在 panel detection 之前做一层可开关预处理：

1. 黑白稿：灰度化、对比度增强、轻度二值化、去噪
2. 彩页：保留彩色原图，同时生成一张边界增强图供 panel detection 使用
3. 对有明显页边留白的图，先做 page border 裁切，减少误检

推荐在这一阶段输出：

```json
{
  "page_id": "page_0001",
  "reading_direction": "rtl",
  "panels": [
    {
      "panel_id": "p001",
      "bbox": [108, 92, 948, 730],
      "polygon": [[108,92],[948,92],[948,730],[108,730]],
      "crop_path": "runtime/panels/page_0001/p001.png",
      "layout_order": 1
    }
  ]
}
```

这个 layout_order 只是第一轮候选顺序，不要直接当最终故事顺序。

## 2. 分镜排序：阅读顺序不等于故事顺序

你的需求强调的是“按故事连续性完成分镜排序”，这比普通的 panel reading order 难一层。

建议把排序拆成两步：

### 第一步：几何候选顺序

使用以下信息生成初始候选：

1. comfyui_panels 给出的 panel order
2. 页面阅读方向，日漫常见为 rtl，国漫 / 条漫素材可能是 ltr 或纵向
3. bbox 的空间关系
4. 对话框尾巴方向和跨分镜人物朝向

这一层解决的是“人通常会按什么顺序读”。

### 第二步：剧情连续性重排

使用 Qwen-VL 对候选顺序做 rerank。输入不只给单张分镜，而是给每个分镜的结构化摘要：

- 角色列表
- 动作
- 场景
- 对话 / 旁白 OCR 结果
- 情绪变化
- 时间 / 地点线索
- 与上一分镜的衔接提示

然后让模型回答两个问题：

1. 哪个顺序最符合剧情推进
2. 哪些分镜虽然空间上相邻，但更像回忆、插叙、同一动作的补镜

建议输出：

```json
{
  "page_id": "page_0001",
  "candidate_orders": [
    ["p001", "p002", "p003", "p004"],
    ["p001", "p003", "p002", "p004"]
  ],
  "best_story_order": ["p001", "p003", "p002", "p004"],
  "reason": "p003 延续了 p001 的动作和对白，p002 更像插入反应镜头"
}
```

这一步是全链路里最依赖 VLM 推理的部分，单靠 panel detection 做不到。

## 3. 分镜内容识别：先快后准

### 推荐资源 1：ComfyUI-Florence2

资源：

- https://github.com/kijai/ComfyUI-Florence2

这个节点适合做“第一轮快速理解”。仓库说明中明确支持：

1. image captioning
2. object detection
3. segmentation
4. Document Visual Question Answering

对漫画分镜来说，它最有价值的不是写华丽描述，而是先把每张分镜变成稳定的中间语义：

- 谁在画面里
- 看到了什么物体
- 是否有文字区域
- 是否能回答“这张图里谁在说话、场景像哪里、有哪些关键动作”

DocVQA 对纯漫画气泡字未必稳，但对印刷清晰、对话框整洁的页面仍然有帮助。

### 推荐资源 2：ComfyUI-QwenVL

资源：

- https://github.com/1038lab/ComfyUI-QwenVL
- https://deepwiki.com/1038lab/ComfyUI-QwenVL/4.6-example-workflows

这个节点适合做“第二轮强理解”。已查到的能力包括：

1. 支持 Qwen2.5-VL 和 Qwen3-VL
2. 有标准节点和高级节点
3. 支持图像和视频输入
4. 自带 example_workflows
5. 支持 preset prompt 和 custom prompt

DeepWiki 的示例工作流里，已经明确给出：

- Basic Image Captioning Workflow
- Prompt Comparison Workflow
- Custom Prompt Refinement Workflow

这非常适合漫画分镜，因为你可以先用通用 prompt 得到描述，再用自定义 prompt 做二次追问，例如：

- 这张分镜里主角正在做什么
- 哪些人可能与上一张分镜是同一角色
- 这句对白更像冲突开端、承接还是结果

### 建议的双阶段识别策略

1. Florence-2 跑首轮摘要
2. Qwen-VL 读取首轮摘要和原图，再输出结构化故事信息

单个分镜建议至少产出以下字段：

```json
{
  "panel_id": "p003",
  "caption": "角色A转头看向门口，角色B在背景中举起手臂",
  "characters": [
    {
      "local_id": "c1",
      "gender_guess": "unknown",
      "age_guess": "teen",
      "hair": "short black",
      "clothes": "school uniform",
      "visible_face": true
    }
  ],
  "actions": ["turning_head", "raising_hand"],
  "speech_text": ["你终于来了"],
  "scene": "classroom doorway",
  "mood": "tense",
  "continuity_cues": {
    "likely_before": "conversation start",
    "likely_after": "response shot"
  }
}
```

后续所有“排序”和“角色归并”都依赖这份结构化结果。

## 4. 跨图角色一致性识别

这一项最容易混淆。它实际分成两件事：

### A. 识别任务

判断分镜 A 里的角色和分镜 B 里的角色是不是同一个人。

### B. 生成任务

后续如果用文生图 / 图生图继续扩展剧情，如何尽量保持角色不变脸。

这两个任务不能混为一谈。InstantID 和 IPAdapter FaceID 更偏 B，不是严格意义上的 A。

### 推荐资源 1：ComfyUI_FaceAnalysis

资源：

- https://github.com/cubiq/ComfyUI_FaceAnalysis

这个仓库的 README 写得很直接：它是用 Dlib 或 InsightFace 来做 face similarity 评估的 ComfyUI 扩展，可用于比较两张脸是否相似。

这正好适合作为“角色识别”里的第一条强特征，尤其适用于：

- 真人风漫画
- 写实风角色
- 面部清晰、正脸或半侧脸较多的页面

它的边界也很明确：

- 看不到脸时没法用
- 夸张漫画风、Q 版、极端角度时稳定性会下降
- 它解决的是脸，不是整个人设一致性

### 推荐资源 2：ComfyUI InstantID

资源：

- https://github.com/cubiq/ComfyUI_InstantID

这个仓库更适合“后续生成保持同脸”。它的特征包括：

1. 原生 InstantID 支持
2. 依赖 antelopev2 的 InsightFace 模型
3. SDXL only
4. 支持 Multi-ID，但作者也明确说 workflow 比较复杂

它适合用在下面场景：

- 当你已经确认某个角色是谁
- 准备用该角色参考图继续生成新分镜
- 希望后续出图时锁定角色面部身份

它不适合作为唯一的“跨分镜识别器”，因为它重点是 condition generation，不是建立角色聚类。

### 推荐资源 3：IPAdapter / FaceID

资源：

- https://github.com/comfyorg/comfyui-ipadapter

这个仓库本体是 IPAdapter 的 ComfyUI 参考实现，README 明确包含：

- FaceID 模型
- FaceID LoRA
- example workflows
- composition / style transfer

它的价值在于：

1. 后续生成时能把参考人脸、人设、风格和构图部分拆开控制
2. 对“角色长得像原角色但姿态和场景变了”这类任务很合适
3. 对长序列生成比单纯文本提示更稳

但它仍然偏生成一致性，不是最终识别裁判。

### 真正可落地的角色归并方案

建议做一个角色注册表，而不是每次都让模型临时判断。

角色注册表字段建议包括：

```json
{
  "character_id": "char_001",
  "canonical_name": "角色A",
  "aliases": ["前辈", "队长"],
  "face_refs": ["runtime/characters/char_001/face_01.png"],
  "face_embedding_refs": ["..."],
  "appearance_traits": {
    "hair": ["short black"],
    "clothes": ["school uniform"],
    "accessories": ["glasses"]
  },
  "scene_history": ["classroom", "corridor"]
}
```

归并时使用三类特征：

1. 人脸相似度：FaceAnalysis / InsightFace
2. 视觉属性相似度：头发、服装、配饰、武器、体型、色彩
3. 叙事线索相似度：名字、称呼、对白承接、位置关系

推荐规则：

1. 有清晰脸时，优先走 face similarity
2. 没脸或脸太小，走视觉属性 + 上下文推理
3. 两者都不稳时，保留 uncertain，不要强行合并

## 5. 推荐的 ComfyUI 实施形态

### 方案 A：纯 ComfyUI 节点编排

适合先做验证版，链路如下：

1. Load Image
2. 预处理节点
3. comfyui_panels 做 panel detection
4. 输出 crop / mask / polygon
5. Florence-2 节点做首轮 caption / detection / DocVQA
6. QwenVL 节点做二轮结构化理解
7. FaceAnalysis 节点做 face similarity
8. 保存 JSON

优点：

- 一切都在 ComfyUI 里
- 易于可视化调试

缺点：

- “多分镜全局重排”和“角色注册表更新”这类状态型逻辑不太优雅

### 方案 B：ComfyUI 负责视觉节点，外部脚本负责编排

这更符合工程化落地，建议作为正式版方向。

分工建议：

1. ComfyUI：切分、视觉理解、embedding 生成
2. 外部 Python 编排：排序、角色注册表、缓存、导出

当前仓库里已经有一个现成的“LM Studio 驱动 ComfyUI API workflow”的工程基座：

- [agent-projects/lmstudio-comfyui-benchmark/README.md](../../agent-projects/lmstudio-comfyui-benchmark/README.md)

这个项目本来是做模型测评的，但它已经证明了三件事：

1. 可以稳定调用 ComfyUI API workflow
2. 可以把 LLM 输出注入 ComfyUI
3. 可以做 checkpoint、续跑、日志和批处理

如果后面要正式做“漫画分镜理解流水线”，建议新开独立项目，复用这套 API 编排思路，而不是把复杂状态硬塞进单个 ComfyUI 图里。

## 6. 当前最值得优先尝试的现成资源

### 第一优先级

1. comfyui_panels
2. ComfyUI-Florence2
3. ComfyUI-QwenVL

理由：这三者基本能把“切割 + 单图理解 + 上下文重排”先跑通。

### 第二优先级

1. ComfyUI_FaceAnalysis
2. ComfyUI_InstantID
3. comfyui-ipadapter

理由：这三者主要解决“角色识别增强”和“后续生成一致性增强”。

## 7. 按当前机器配置的模型建议

本机假设是 RTX 5070 Ti 级别、16GB VRAM。按现有资源，建议从以下组合起步：

### 视觉理解模型

1. Florence-2-base 或 Florence-2-large：做首轮描述和 DocVQA
2. Qwen3-VL-4B-Instruct 8-bit：做二轮故事理解和排序

### 角色特征模型

1. InsightFace antelopev2：做人脸相似度和 InstantID 依赖
2. 必要时再引入 FaceID LoRA 或 IPAdapter FaceID

为什么不建议一开始就上更大的 VLM：

1. 页级批处理时显存占用会很快堆高
2. 你后面还要保留 panel crops、caption、embedding 缓存
3. 先把链路跑通，比先追求最大模型更重要

## 8. MVP 落地顺序

建议按 4 个阶段推进：

### 阶段 1：分镜切割 MVP

目标：

- 给整页漫画输出 panel crop 和 layout_order

验收标准：

- 规则页面切分正确率足够高
- 黑白和彩页都能出结果

### 阶段 2：单图理解 MVP

目标：

- 每个分镜输出 caption、角色候选、动作、对话文本、场景

验收标准：

- 至少能导出结构化 JSON

### 阶段 3：角色归并 MVP

目标：

- 给同页或同章节中的分镜分配稳定 character_id

验收标准：

- 清晰正脸场景下，同角色匹配稳定
- 模糊场景允许 uncertain，不强判

### 阶段 4：剧情连续性排序 MVP

目标：

- 在 panel layout order 之上，输出 best_story_order 和分镜故事摘要

验收标准：

- 对常见对话页、动作承接页，排序优于纯几何顺序

## 9. 风险点

1. 同一页里存在跨页大图、叠层分镜、无边框分镜时，panel detection 会明显变难。
2. 纯漫画风角色经常出现侧脸、背影、遮挡，单靠 FaceAnalysis 不够。
3. 对白 OCR 在手写字、特效字、拟声词很多时会不稳定。
4. “故事连续性”本身不是完全客观问题，建议保留候选顺序和理由，不要只输出一个硬编码答案。
5. 现成节点多数擅长单图，不擅长维护长篇章节级上下文，所以必须保留外部状态文件。

## 10. 最终建议

如果只选一条最稳的落地路线，建议这样开始：

1. 用 comfyui_panels 做分镜切割和基础顺序
2. 用 Florence-2 做每个分镜的首轮 caption / 文本问答
3. 用 QwenVL 基于所有分镜摘要做剧情连续性重排和故事串联
4. 用 FaceAnalysis 建角色注册表
5. 需要继续生成连续画面时，再接 InstantID 或 IPAdapter FaceID

这条路线的优点是：

- 每一步都有现成 ComfyUI 资源可依赖
- 真正困难的地方被集中到“排序逻辑”和“角色注册表”两块，便于单独攻关
- 后续既能做理解，也能接生成一致性，不会推倒重来

如果后续决定正式开做，建议把“状态管理、缓存、导出、角色注册表”放进一个独立项目，不要全部堆在单个 ComfyUI workflow 里。
