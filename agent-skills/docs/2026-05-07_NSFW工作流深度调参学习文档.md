## 1. NSFW 多轮生成效果衰减：原因与解决思路

### 1.1 问题现象

使用同一套参数（seed=777, steps=20, cfg=5.0, shift=8.0, LoRA strength=0.8），多轮重复生成后，或调高生成步数 / 帧数后，NSFW 内容效果明显退化——画面趋向保守、内容"变安全"、细节缺失。

### 1.2 根因分析（按影响程度排序）

#### 原因一：CFG 值过高 → 基础模型"安全偏置"被放大 🔴 最可能

**机制**：Wan 2.2 基础模型是内容对齐模型（content-aligned），内部存在隐式的安全偏置。CFG（Classifier-Free Guidance）的公式为：

$$\text{output} = \text{uncond} + \text{cfg} \times (\text{cond} - \text{uncond})$$

当 cfg=5.0 时，有条件预测与无条件预测的差值被放大 5 倍。但 **LoRA 的修改只作用于有条件分支（cond）**，无条件分支仍然走基础模型的"安全"路径。差值放大后，基础模型的安全偏置会通过无条件分支间接渗透到最终输出。

**证据**：NSFW 社区实践通常建议 Wan2.2 的 CFG 在 1.0~3.0 之间，5.0 明显偏高。

**解决**：将 CFG 从 5.0 降到 **1.5~3.0**，观察 NSFW 效果变化。低 CFG 会让模型更"自由"，LoRA 的相对影响力更大。

---

#### 原因二：shift 值过高 → 细节阶段被压缩 🔴 很可能

**机制**：Wan2.2 使用 Flow Matching 调度器，shift 参数通过以下公式重映射噪声水平：

$$\sigma_{\text{shifted}} = \frac{\text{shift} \times \sigma}{1 + (\text{shift} - 1) \times \sigma}$$

- **低 σ（后期，细节阶段）**：分母 ≈ 1，$\sigma_{\text{shifted}} \approx \text{shift} \times \sigma$，被线性放大
- **高 σ（早期，结构阶段）**：分母 ≈ shift × σ，$\sigma_{\text{shifted}} \approx 1$，被压缩到接近 1

**实际效果**：shift=8.0 时，绝大部分采样步被分配给早期结构建立阶段，后期细节填充阶段被严重压缩。而 **NSFW LoRA 的效果主要在细节阶段体现**（皮肤纹理、身体细节等），导致 LoRA 发挥作用的时间窗口被挤压。

**数值对比**（以 20 步为例）：

| shift | 早期步（结构） | 后期步（细节） | NSFW 效果 |
|-------|-------------|-------------|----------|
| 3.0   | ~8 步       | ~12 步      | 细节充分 |
| 5.0   | ~12 步      | ~8 步       | 平衡     |
| 8.0   | ~16 步      | ~4 步       | 细节不足 |
| 12.0  | ~18 步      | ~2 步       | 严重退化 |

**解决**：将 shift 从 8.0 降到 **3.0~5.0**（Wan2.2 默认推荐 5.0），分配给细节阶段更多步数。

---

#### 原因三：固定 seed 多轮重复 → 收敛到同一局部最优 🟡 可能

**机制**：固定 seed=777 使每轮生成的初始噪声完全相同。增加步数后，模型有更多迭代机会朝着基础模型的"安全"分布收敛。基础模型的训练数据以安全内容为主，在更多步数下，模型会更彻底地趋向训练分布的中心。

**解决**：
- 每轮使用不同 seed（设为 randomize / -1）
- 或者减小步数变化幅度，每轮只微调 2~4 步

---

#### 原因四：fp8 量化精度损失随步数累积 🟡 可能

**机制**：当前使用 `wan2.2_t2v_high_noise_14B_fp8_scaled.safetensors`（fp8 量化）。虽然 fp8_scaled 使用了逐通道缩放来减少精度损失，但在 20+ 步的迭代采样中，每步的量化误差会累积。LoRA 权重同样受 fp8 精度限制。步数越多，累计误差越大，LoRA 的精细调节效果被量化噪声淹没。

**解决**：
- 如显存允许，尝试 bf16 版本模型（约 28GB，超出 16GB VRAM，需要 offload）
- 或在当前 fp8 下将步数控制在 15~25 步
- 使用 lightx2v 类加速 LoRA 来减少所需步数

---

#### 原因五：帧数增加 → 注意力分散 🟢 次要

**机制**：从 81 帧增加到 121/161 帧时，模型的时空自注意力窗口内的 token 数增加。模型需要在更多帧之间分配"注意力预算"，对单帧的细节关注度下降。NSFW 内容通常依赖局部细节，帧数越多，局部细节信号越容易被全局平均稀释。

**解决**：
- 如果不需要长视频，保持 81 帧（~5 秒@16fps）
- 需要长视频时，使用分段生成 + Pusa 扩展，而非单次生成

---

### 1.3 推荐的修复参数组合

| 参数 | 当前值（有问题） | 推荐值 | 说明 |
|------|---------------|--------|------|
| cfg | 5.0 | **2.0 ~ 3.0** | 最关键修改 |
| shift | 8.0 | **4.0 ~ 5.0** | 次关键修改 |
| steps | 20 | **15 ~ 20** | 配合 shift 调整 |
| seed | 777（固定） | **randomize** | 避免重复收敛 |
| LoRA strength | 0.8 | **0.75 ~ 0.9** | 适度微调 |
| frames | 81 | **81**（保持） | 不盲目增加 |

**预期效果**：调整 cfg 和 shift 后，NSFW 效果应恢复到可接受水平。如果仍不理想，再检查 LoRA 是否在正确的模型阶段加载（high_noise vs low_noise）。

## 2. LoRA 加载器参数详解

### 2.1 LoRA 是什么

LoRA（Low-Rank Adaptation）是一种高效的模型微调技术。它不修改原始模型权重，而是在模型旁路添加低秩矩阵（A 和 B），训练时只更新这些低秩矩阵。

### 2.2 模型强度（strength）的数学原理

从本机 `custom_linear.py` 的源码实现来看，LoRA 的应用公式为：

**标准 LoRA（双矩阵）**：
```python
patch_diff = lora_B @ lora_A              # 低秩乘积得到全尺寸增量
alpha = lora_alpha / rank                  # 缩放因子
scale = lora_strength * alpha              # 最终缩放
new_weight = base_weight + patch_diff * scale
```

**单文件 LoRA（如 NSFW-22-H-e8.safetensors，已合并为单一增量）**：
```python
new_weight = base_weight + lora_diff * lora_strength
```

### 2.3 strength 正负值的含义

#### 正值（0.0 ~ 1.0+）

| strength 值 | 效果 | 说明 |
|------------|------|------|
| **0.0** | LoRA 完全无效 | 等于只使用基础模型 |
| **0.3 ~ 0.5** | 轻微效果 | LoRA 特征隐约可见，基础模型风格占主导 |
| **0.6 ~ 0.8** | 中等效果（推荐） | LoRA 特征清晰但不过度，适合大多数场景 |
| **0.9 ~ 1.0** | 完整效果 | LoRA 训练时的标准强度 |
| **> 1.0** | 过饱和 | LoRA 特征被过度放大，可能导致画崩、畸变 |

**NSFW LoRA 建议**：从 0.7 开始，逐步增加到 0.85，找到"效果好但不崩"的平衡点。

#### 负值（-1.0 ~ 0.0）

| strength 值 | 效果 | 说明 |
|------------|------|------|
| **-0.3 ~ -0.1** | 轻微反向 | 略微抑制 LoRA 学到的特征 |
| **-0.8 ~ -0.5** | 中等反向 | 明显反向 LoRA 特征，可用于"去风格化" |
| **-1.0** | 完全反向 | 基础模型权重减去 LoRA 的全部增量 |

**负值应用场景**：
- **去除特定风格**：如果某个 LoRA 学的是"动漫风格"，负值可以反向推动画面更写实
- **修正过强 LoRA**：如果 LoRA 效果过强导致崩图，用小幅负值做"减法"
- **实验性混合**：用正值的角色 LoRA + 负值的风格 LoRA 组合创造新效果

> ⚠️ **警告**：负值不是"反向 NSFW"，而是从基础模型中减去 LoRA 学到的权重偏移。对于 NSFW LoRA（学到的是人体细节/裸露特征），负值通常会让人体变得不自然、解剖结构异常。不推荐对 NSFW LoRA 使用负值。

### 2.4 多 LoRA 叠加规则

当使用 `WanVideoLoraSelectMulti` 加载多个 LoRA 时：

- **merge_loras = false**（默认）：每个 LoRA 独立作为一个增量，按 `strength_0 + strength_1 + ...` 顺序叠加：
  $$\text{weight} = \text{base} + \Delta_0 \times s_0 + \Delta_1 \times s_1 + \dots$$

- **merge_loras = true**：多个 LoRA 先合并为一个增量再应用，可减少计算开销但灵活性降低

**叠加顺序影响**：先应用的 LoRA 效果可能被后应用的 LoRA 部分覆盖。如果同时使用加速 LoRA（lightx2v）和 NSFW LoRA：
- 建议加速 LoRA 放前面（strength 0.99）
- NSFW LoRA 放后面（strength 0.7~0.85）

## 3. K采样器 / WanVideoSampler 核心参数详解

### 3.1 K采样器是什么

在扩散模型中，K采样器（K-Sampler）负责执行**去噪采样过程**（reverse diffusion / denoising）。它的任务是从随机噪声出发，通过多步迭代逐步去除噪声，最终生成清晰的图像或视频。

### 3.2 WanVideoSampler 参数详解

本机 Wan 2.2 工作流使用 `WanVideoSampler` 节点（而非标准 KSampler），其核心参数如下：

#### steps（采样步数）

| 属性 | 说明 |
|------|------|
| 含义 | 去噪迭代的总步数 |
| 默认值 | 30（WanVideoSampler）/ 20（本机 T2V 工作流） |
| 实际范围 | 10 ~ 50 |

**效果**：
- **少步数（10~15）**：生成速度快，但细节不足，可能出现噪点或模糊
- **中步数（15~25）**：性价比最高的区间，配合 lightx2v LoRA 可降到 8~12 步
- **多步数（30~50）**：细节更精细，但收益递减，且量化模型（fp8）的误差累积更明显

**与 NSFW 的关系**：步数越多，基础模型的"安全偏置"有更多迭代机会表达，NSFW 效果可能反而衰减。

#### cfg（Classifier-Free Guidance 引导强度）

| 属性 | 说明 |
|------|------|
| 含义 | 模型遵循提示词的程度 |
| 默认值 | 6.0（WanVideoSampler）/ 5.0（本机 T2V） |
| 实际范围 | 1.0 ~ 10.0 |

**效果曲线**：

| CFG 值 | 出图特征 | 推荐场景 |
|--------|---------|---------|
| 1.0 ~ 2.0 | 自由创作，可能偏离提示词 | 图生视频、姿态驱动 |
| 2.0 ~ 4.0 | 平衡提示词遵循与创造性 | **NSFW 推荐区间** |
| 4.0 ~ 6.0 | 严格遵循提示词，但可能过饱和/伪影 | 精确控制场景 |
| 6.0 ~ 10.0 | 高对比度、色彩过饱和、可能崩图 | 不推荐 |

**原理**：CFG 通过计算"有条件预测"与"无条件预测"的加权差值来引导生成方向：

$$\text{output} = \text{uncond} + \text{cfg} \times (\text{cond} - \text{uncond})$$

- cfg=1.0 时，output = cond（纯有条件），最自由
- cfg 越高，(cond - uncond) 的差值被放大，模型越"听话"
- 但过高的 cfg 会放大噪声和伪影

> ⚠️ **NSFW 关键认知**：基础 Wan 模型的"无条件预测"天然偏向安全内容。高 CFG 虽然强化了提示词遵循，但差值计算中"无条件"分支的安全偏置也会被放大，反而抑制 NSFW 效果。推荐 CFG **2.0 ~ 3.0**。

#### shift（噪声调度偏移）

| 属性 | 说明 |
|------|------|
| 含义 | 控制噪声水平（sigma）在不同采样步之间的分布 |
| 默认值 | 5.0 |
| 实际范围 | 1.0 ~ 15.0 |

**数学原理**（Flow Matching 调度器）：

$$\sigma_{\text{shifted}} = \frac{\text{shift} \times \sigma}{1 + (\text{shift} - 1) \times \sigma}$$

其中 σ 是原始的均匀分布的噪声水平（0~1）。

**直观理解**：

| shift 值 | 早期步（高噪声） | 后期步（低噪声） | 适用场景 |
|----------|---------------|---------------|---------|
| 1.0 ~ 3.0 | 较少预算 | 较多预算（细节好） | 静态图、精细细节 |
| 3.0 ~ 5.0 | 均衡 | 均衡 | 通用推荐 |
| 5.0 ~ 8.0 | 较多预算 | 较少预算 | 大幅运动视频 |
| 8.0 ~ 15.0 | 大量预算 | 很少预算 | 快速动作、镜头切换 |

**选择建议**：
- 静态/慢速场景：shift 3.0~5.0（确保细节）
- 快速运动场景：shift 5.0~8.0（确保动作流畅）
- NSFW 内容：shift 4.0~5.0（平衡结构与细节）

#### scheduler（调度器）

WanVideoWrapper 支持的调度器及其特点：

| 调度器 | 速度 | 质量 | 推荐场景 |
|--------|------|------|---------|
| **unipc** | ★★★★ | ★★★★★ | **首选推荐**，速度快质量好 |
| **dpm++** | ★★★ | ★★★★★ | 最高质量，适合最终出图 |
| **euler** | ★★★★★ | ★★★★ | 最快，适合快速测试 |
| **lcm** | ★★★★★ | ★★★ | 极速（4~8步），需 LCM LoRA |
| **deis** | ★★★★ | ★★★★ | 均衡选择 |
| **res_multistep** | ★★★ | ★★★★★ | 高质量，大分辨率稳定 |

**NSFW 推荐**：unipc 或 dpm++，15~20 步。

#### seed（随机种子）

- **固定值**：可复现，但多轮重复可能导致收敛到同一"安全"分布
- **randomize（-1）**：每轮不同噪声起点，结果多样
- **增量调整**：如 seed=777, 778, 779... 做微调变体

#### force_offload（强制卸载）

- **true**：每轮采样后释放模型到 CPU/offload 设备，释放 VRAM
- **false**：模型留在 GPU，下一轮更快但占用 VRAM

16GB VRAM 下建议 **true**，避免多轮累积 OOM。

#### denoise_strength（去噪强度，图生视频/V2V）

- **1.0**：从纯噪声开始（T2V 模式）
- **0.5~0.8**：保留部分原始视频结构（V2V 模式）
- **0.0~0.3**：仅做微调

### 3.3 正/负提示词与 K采样器的协同机制

**正向提示词（Positive Prompt）**：
- 被 CLIP/T5 编码为文本嵌入（text embeddings）
- 在采样每一步中作为"条件信号"注入模型
- 描述 **想要生成的内容**

**负向提示词（Negative Prompt）**：
- 同样被编码为文本嵌入
- 在采样每一步中作为"无条件信号"（或独立负条件）
- 描述 **不想要的内容**

**协同公式**（CFG 模式下）：
$$\text{output} = \text{model}(\text{noisy_latent}, \text{neg_embeds}) + \text{cfg} \times [\text{model}(\text{noisy_latent}, \text{pos_embeds}) - \text{model}(\text{noisy_latent}, \text{neg_embeds})]$$

这意味着模型同时运行两次推理（一次带正提示词，一次带负提示词），然后按 CFG 权重组合结果。这也是为什么高 CFG 会消耗更多计算资源。

## 4. 双 CLIP / VAE / 双 UNet 加载器用途详解

### 4.1 CLIP 加载器（文本编码器）

#### 单个 CLIP 加载器（当前 T2V 工作流实际使用）

本机 Wan2.2 T2V 工作流使用 **一个** CLIP 加载器（`CLIPLoader` 或 `LoadWanVideoT5TextEncoder`）：

| 参数 | 值 | 说明 |
|------|-----|------|
| clip_name / t5_model | `umt5_xxl_fp8_e4m3fn_scaled.safetensors` | Google UMT5 XXL 文本编码器 |
| type | `wan` | 指定为 Wan 兼容模式 |

**作用**：将自然语言提示词转换为模型可理解的**文本嵌入向量**（text embeddings）。UMT5 是一个多语言 T5 变体，支持中英文混合提示词。

#### 什么时候需要双 CLIP？

在某些高级架构中（如 SD3、Flux），会同时使用两个 CLIP 编码器：

| 编码器 | 参数量 | 作用 |
|--------|--------|------|
| **CLIP-L** (OpenAI) | ~430M | 提供视觉-语义对齐（擅长物体/场景描述） |
| **T5-XXL** (Google) | ~11B | 提供深度语言理解（擅长复杂指令/关系） |

**双 CLIP 的工作方式**：
1. 两个编码器独立编码同一段提示词
2. 输出向量沿特征维度拼接（concatenate）
3. 模型在推理时同时关注两种语义表示

**Wan 2.2 的情况**：Wan 2.2 只使用 **UMT5 XXL** 一个文本编码器，不需要双 CLIP。如果你在 ComfyUI 节点列表中看到"双 CLIP"的提及，那可能是针对 SD3/Flux/AuraFlow 等工作流的。

---

### 4.2 VAE 加载器（变分自编码器）

| 参数 | 值 | 说明 |
|------|-----|------|
| vae_name | `Wan2_1_VAE_bf16.safetensors` | Wan 2.1 3D VAE |
| precision | `bf16` | 半精度浮点 |

#### VAE 的核心作用

VAE（Variational Autoencoder）是像素空间与潜空间之间的**桥梁**：

```
像素空间 (RGB, 512×768×3)  ←→  潜空间 (Latent, 64×96×16)
       ↑ 编码 (Encode)                ↑ 解码 (Decode)
```

**为什么需要 VAE？**

| 维度 | 像素空间 | 潜空间 |
|------|---------|--------|
| 分辨率 | 512×768×3 = 1.18M 值/帧 | 64×96×16 = 98K 值/帧 |
| 压缩比 | 1× | ~12× 空间压缩 |
| 计算量 | 极高 | 可管理 |

**Wan 2.1 VAE 的特殊性（3D VAE）**：
- 不仅压缩空间（宽×高），还压缩时间维度
- 压缩比：时间 4×、空间 8×8 = 总共 256×
- 81 帧的 512×768 视频 → 潜空间约 21×64×96×16

**只有解码时才回到像素空间**：整个采样过程都在潜空间中进行，只有最终输出时用 VAE Decode 还原为可视画面。

### 4.3 双 UNet 加载器（Wan 2.2 I2V 两阶段架构）

这是 Wan 2.2 **图生视频（I2V）** 工作流的独特设计。

#### 为什么需要两个 UNet？

Wan 2.2 I2V 14B 将视频生成拆分为两个专业化的 UNet：

| UNet | 模型文件 | 负责阶段 | 噪声水平 |
|------|---------|---------|---------|
| **Low Noise UNet** | `wan2.2_i2v_low_noise_14B_fp8_scaled.safetensors` | 第一阶段（早期） | 低噪声 → 建立骨架 |
| **High Noise UNet** | `wan2.2_i2v_high_noise_14B_fp8_scaled.safetensors` | 第二阶段（后期） | 高噪声 → 填充细节 |

#### 两阶段协作流程

```
随机噪声
  │
  ▼
┌─────────────────────────────┐
│  KSampler #1 (Low Noise)    │  ← low_noise UNet
│  add_noise = enable          │  ← lightx2v LoRA (low_noise版)
│  start=0, end=3, steps=8    │
│  return_with_leftover_noise  │  建立视频大致结构和运动骨架
│  = enable                    │
└──────────┬──────────────────┘
           │ 带剩余噪声的 latent
           ▼
┌─────────────────────────────┐
│  KSampler #2 (High Noise)   │  ← high_noise UNet
│  add_noise = disable         │  ← lightx2v LoRA (high_noise版)
│  start=3, end=6, steps=6    │  ← NSFW LoRA（仅挂载在此）
│  return_with_leftover_noise  │  填充纹理、皮肤、NSFW细节
│  = disable                   │
└──────────┬──────────────────┘
           │
           ▼
       VAE Decode → 最终视频
```

#### 关键设计要点

**add_noise 的衔接机制**：
- 第一阶段 `add_noise=enable`：从纯随机噪声开始
- 第一阶段 `return_with_leftover_noise=enable`：不输出完全去噪的干净 latent，而是保留一定噪声
- 第二阶段 `add_noise=disable`：不添加新噪声，直接从第一阶段残留噪声继续
- 这保证了两个阶段之间的 **无缝衔接**

**NSFW LoRA 只挂 High Noise UNet**：
从本机工作流配置来看，NSFW LoRA (`NSFW-22-H-e8.safetensors`, strength=0.75) 只连接在 High Noise UNet 之后。这是因为：
- NSFW 内容主要涉及皮肤纹理、身体细节 → 这些是"细节阶段"的工作
- High Noise UNet 专门负责细节填充
- 如果挂在 Low Noise UNet，可能在结构阶段引入不需要的偏差

**lightx2v 加速 LoRA 各挂一个**：
- Low Noise 版 lightx2v → 加速第一阶段
- High Noise 版 lightx2v → 加速第二阶段
- 两版必须配对使用，不能混用

#### Wan 2.2 T2V 工作流（文生视频）只有单 UNet

本机的 T2V NSFW 工作流只使用 `wan2.2_t2v_high_noise_14B_fp8_scaled.safetensors` 一个模型文件，不需要双 UNet。

## 6. 调参实战技巧

### 6.1 NSFW 出图质量调优流程

```
第1步：固定 seed=777，快速验证基线
  ├── 参数：steps=15, cfg=2.5, shift=5.0, LoRA=0.8
  └── 目标：确认基础效果能用

第2步：调整 CFG（最关键）
  ├── 从 2.0 到 4.0，每次 +0.5
  ├── 对比 NSFW 内容的明显程度
  └── 找到"最明显但不崩"的 CFG 值

第3步：调整 shift
  ├── 从 3.0 到 7.0，每次 +1.0
  ├── 观察运动流畅度 vs 细节清晰度
  └── 运动场景用 5~7，静态场景用 3~5

第4步：微调 LoRA strength
  ├── 从 0.65 到 0.95，每次 +0.05
  └── 找到"风格到位但不过度"的点

第5步：放开 seed 随机化
  ├── 用上一步最优参数
  └── seed = randomize，生成 5~10 个变体验证

第6步：增加 steps（如需更高画质）
  ├── 每次 +5 步，到 25~30 步
  └── ⚠️ 注意 NSFW 效果是否随步数衰减
```

### 6.2 快速诊断表

| 症状 | 可能原因 | 检查方向 |
|------|---------|---------|
| NSFW 效果弱/没有 | CFG 过高 | 降到 2.0~3.0 |
| NSFW 效果弱/没有 | shift 过高 | 降到 4.0~5.0 |
| NSFW 效果弱/没有 | LoRA 未正确加载 | 检查 LoRA 连线 |
| 画面模糊/噪点 | steps 太少 | 增加到 20+ |
| 画面模糊/噪点 | shift 太低 | 提高到 5.0+ |
| 动作不流畅/闪烁 | CFG 过高 | 降到 1.5~2.5 |
| 动作不流畅/闪烁 | shift 太低 | 提高到 6.0+ |
| 多轮后效果退化 | 固定 seed | 改用 randomize |
| 多轮后效果退化 | 量化误差累积 | 减少 steps 或换 bf16 |
| 角色外观不一致 | LoRA 过强 | 降到 0.5~0.7 |
| 角色外观不一致 | 提示词矛盾 | 精简提示词 |
| 颜色过饱和 | CFG 过高 | 降到 3.0 以下 |
| 显存不足 OOM | force_offload=false | 改为 true |
| 显存不足 OOM | frames 过多 | 减少帧数或分段生成 |

### 6.3 本机硬件适配建议（RTX 5070 Ti 16GB VRAM）

| 分辨率 | 帧数 | steps | 预估 VRAM | 建议 |
|--------|------|-------|-----------|------|
| 512×768 | 57 | 20 | ~12 GB | ✅ 安全 |
| 512×768 | 81 | 20 | ~14 GB | ✅ 可行 |
| 512×768 | 81 | 30 | ~15 GB | ⚠️ 可能 OOM |
| 832×480 | 81 | 20 | ~13 GB | ✅ 安全 |
| 832×480 | 121 | 20 | ~15 GB | ⚠️ 接近上限 |
| 1280×720 | 81 | 20 | >16 GB | ❌ 需要 offload/tiling |

**显存优化策略**：
1. 保持 `force_offload=true`
2. 使用 fp8 量化模型（已在使用）
3. 使用 lightx2v 加速 LoRA（减少所需步数）
4. 先小分辨率验证参数，再放大
5. 使用 `attention_mode=sdpa`（比 flash_attn 省显存）

### 6.4 长视频分段生成策略

对于需要 >81 帧的长视频：

```
第1段（帧 0~69）   ──→ 输出视频 A
第2段（帧 57~126） ──→ 输出视频 B（与A重叠12帧）
第3段（帧 114~183）──→ 输出视频 C（与B重叠12帧）
...
```

使用 `continue_motion_max_frames=12` 和 `video_frame_offset` 参数控制分段衔接。最终用视频编辑工具去重叠拼接。
