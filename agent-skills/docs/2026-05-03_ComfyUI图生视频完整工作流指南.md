## 1. 现有工作流节点参数详解

当前工作流 `CMS-26-5-1.json` 是一个典型的 **WanAnimate 姿态驱动图生视频** 工作流。它使用一张静态参考图 + 一段姿态参考视频，驱动角色按照参考视频的动作运动，同时保持参考图的外观一致性。

### 1.1 工作流整体架构

```
参考图(LoadImage) ──┐
                    ├──> WanAnimateToVideo ──> KSamplerAdvanced(low_noise) ──> KSamplerAdvanced(high_noise) ──> VAEDecode ──> CreateVideo ──> SaveVideo
姿态视频(VHS_Load) ──┤
DWPreprocessor(Body)─┤
DWPreprocessor(Face)─┘
```

**双采样器架构说明**：
- **KSamplerAdvanced #86（low_noise 首段）**：使用 `wan2.2_i2v_low_noise_14B` 模型，`add_noise=enable`，`start_at_step=0, end_at_step=3`，先在低噪声阶段生成视频的大致结构和运动骨架。
- **KSamplerAdvanced #85（high_noise 尾段）**：使用 `wan2.2_i2v_high_noise_14B` 模型，`add_noise=disable`，`start_at_step=3, end_at_step=6`，在高噪声阶段填充细节、纹理和最终画面质量。

这是一种典型的 **低噪声→高噪声两阶段级联采样** 策略，能有效提升视频质量。

### 1.2 核心节点详细参数

#### LoadImage (节点 #1) - "Start Image"
| 参数 | 当前值 | 说明 |
|------|--------|------|
| image | `微信图片_20260420101814_11778_48.jpg` | 参考图片路径，来自 `ComfyUI/input/` |
| upload | `image` | 图片模式 |

**作用**：加载一张静态参考图片，作为视频生成的外观来源。WanAnimate 会提取此图片中的角色外观，并将其"驱动"到参考姿态上。

---

#### VHS_LoadVideoFFmpegPath (节点 #124) - "Pose Reference Video"
| 参数 | 当前值 | 说明 |
|------|--------|------|
| video | `D:/ComfyUI-aki-v3/...mp4` | 姿态参考视频的完整路径 |
| force_rate | 16 | 强制帧率，设为16fps |
| custom_width | 512 | 自定义宽度 |
| custom_height | 768 | 自定义高度 |

**作用**：加载一段参考视频（可以是真人舞蹈、动作视频等），作为目标姿态的参考源。输出的图像帧会被送入 DWPreprocessor 提取人体姿态骨架。

---

#### DWPreprocessor #125 - "Body + Hands Pose"
| 参数 | 当前值 | 说明 |
|------|--------|------|
| detect_hand | enable | 启用手部检测 |
| detect_body | enable | 启用身体检测 |
| detect_face | disable | 禁用人脸检测 |
| resolution | 512 | 检测分辨率，越高越精细但越慢 |
| bbox_detector | yolox_l.torchscript.pt | 边界框检测模型 |
| pose_estimator | dw-ll_ucoco_384_bs5.torchscript.pt | 姿态估计模型（DWPose） |
| scale_stick_for_xinsr_cn | disable | 不是 xinsr 控制网络场景，禁用 |

**作用**：提取参考视频中的身体+手部姿态骨架图，作为 WanAnimateToVideo 的 `pose_video` 输入。输出的骨架图决定了生成视频中角色的动作。

---

#### DWPreprocessor #126 - "Face Motion Guide"
| 参数 | 当前值 | 说明 |
|------|--------|------|
| detect_hand | disable | 禁用手部 |
| detect_body | disable | 禁用身体 |
| detect_face | enable | **启用人脸检测** |
| resolution | 512 | 检测分辨率 |

**作用**：提取参考视频中的人脸运动信息（面部关键点），作为 `face_video` 输入。这能确保生成视频中的人脸表情、朝向跟随参考视频。

---

#### WanAnimateToVideo (节点 #98)
| 参数 | 当前值 | 说明 |
|------|--------|------|
| width | 512 | 输出视频宽度 |
| height | 768 | 输出视频高度（竖屏9:16） |

| video_frame_offset | 0 | 视频帧偏移（用于续接已生成的视频） |

**输入连线**：
- `reference_image` ← LoadImage 输出
- `pose_video` ← Body DWPreprocessor 输出
- `face_video` ← Face DWPreprocessor 输出

**作用**：核心编排节点。接收参考图、姿态视频、人脸视频，输出 latent、positive conditioning、negative conditioning 给两个 KSampler。

## 2. 剧情连贯与画面连贯方案

### 2.1 当前工作流的画面连贯机制

工作流 CMS-26-5-1 已经使用了多项确保画面连贯的技术：

| 技术 | 实现节点 | 作用 |
|------|----------|------|
| **姿态驱动** | DWPreprocessor → pose_video | 确保生成角色的动作与参考视频一致 |
| **人脸运动引导** | DWPreprocessor → face_video | 确保面部表情、朝向跟随参考 |
| **参考图锚定** | LoadImage → reference_image | 确保角色外观一致性 |
| **两阶段采样** | 低噪声→高噪声级联 | 先建骨架再填细节，减少帧间抖动 |
| **continue_motion** | WanAnimateToVideo 参数 | 分段生成时维持运动连续性 |

### 2.2 实现剧情连贯的长视频方案

对于需要剧情连贯的长视频，有以下几种策略：

#### 方案 A：分段生成 + 连续性桥接（推荐入门）

1. **剧情分镜规划**：将完整剧情拆分为多个镜头（分镜），每个镜头时长 3-5 秒
2. **每段使用一张关键帧图片**作为 reference_image
3. **使用 continue_motion_max_frames=12** 在段与段之间创建重叠帧
4. **video_frame_offset** 参数用于控制每段在时间轴上的位置
5. 最后用视频编辑软件（如 DaVinci Resolve、Premiere）拼接各段

```
示例剧情拆分：
镜头1（0-3秒）：角色从左侧走入画面 → 使用图片A，设定进入动作
镜头2（2.5-6秒）：角色停下来，转头看向右侧 → 使用图片B，设定转头动作（与镜头1重叠0.5秒）
镜头3（5-8秒）：角色微笑并挥手 → 使用图片C
```

#### 方案 B：SkyReels Diffusion Forcing 扩展（WanVideoWrapper 已支持）

SkyReels 是一种扩散强制技术，可以在生成过程中强制指定帧之间的过渡逻辑。相关文件：
- `ComfyUI/custom_nodes/ComfyUI-WanVideoWrapper/skyreels/nodes.py`
- 示例工作流：`wanvideo_2_1_14B_skyreels_diffusion_forcing_extension_example_01.json`

这种技术允许你定义帧与帧之间的因果依赖关系，适合制作有明确时间线的剧情视频。

#### 方案 C：Pusa 扩展（支持超长视频）

WanVideoWrapper 内置了 Pusa 扩展功能，用于将视频延长到原模型限制之上：
- `wanvideo_2_2_14B_Pusa_extension_example_01.json`
- `wanvideo_2_1_14B_pusa_I2V_example_01.json`

Pusa 通过时间步重映射技术，可以在不增加显存消耗的情况下生成更长视频。

#### 方案 D：Context Window 技术（支持极长视频）

WanVideoWrapper 支持滑动上下文窗口：
- `context_windows/context.py` 实现
- 支持 1025 帧生成，使用 81 帧窗口 + 16 帧重叠
- 示例：1.3B 模型下 1025 帧仅消耗 5GB VRAM

**使用建议**：对于剧情连贯的视频，优先使用方案A（手动分段+拼接），后期可尝试方案B/C/D进行自动化长视频生成。

### 2.3 画面连贯性调优技巧

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 帧间闪烁（flicker） | CFG 过高或采样步数不足 | 降低 CFG 到 1.0-1.4，增加 steps 到 8-12 |
| 角色外观突变 | 参考图锚定不够强 | 确保 reference_image 连接已正确接通 |
| 动作不自然 | 姿态参考视频质量差 | 选择流畅、清晰的动作视频，避免过快运动 |
| 人脸抖动 | face_video 参数不当 | 调整 DWPreprocessor 的 resolution 到 768 |

## 3. 音频与配乐集成方案

### 3.1 ComfyUI 原生音频支持

ComfyUI 核心已内置音频处理能力：

- **CreateVideo 节点**有 `audio` 输入端口（AUDIO 类型），可将音频轨道写入输出视频
- **VHS_LoadVideoFFmpegPath** 节点有 `audio` 输出端口，可直接从视频文件提取音频

**最简单方案**：如果你有一段带音频的参考视频，可以直接用 VHS 节点的 audio 输出连接到 CreateVideo 的 audio 输入：

```
VHS_LoadVideoFFmpegPath.audio → CreateVideo.audio
```

这会将原始参考视频的音频轨道嵌入最终输出视频中。

### 3.2 Ovi - 图生视频+音频联合生成（WanVideoWrapper 已支持）

Ovi 是目前最先进的图生视频+音频方案，已在 ComfyUI-WanVideoWrapper 中实现：
- 节点代码：`ComfyUI/custom_nodes/ComfyUI-WanVideoWrapper/Ovi/nodes_ovi.py`
- 示例工作流：`wanvideo_2_2_5B_Ovi_image_to_video_audio_example_01.json`

Ovi 工作原理：
1. 使用 MMAudio VAE 将音频编码为 latent
2. 音频 latent 与视频生成过程联合推理
3. 视频的运动节奏与音频节奏自动对齐
4. 最终输出带有同步音频的视频

**所需模型**：
- MMAudio VAE（放入 `ComfyUI/models/mmaudio/`）
- BigVGAN vocoder（放入 `ComfyUI/models/vae/`）

**使用限制**：
- 需要额外下载约 2-4 GB 的音频模型
- 音频+视频联合生成会增加约 2-4 GB 显存消耗
- 当前支持 Ovi 的 Wan 版本主要是 5B 模型

### 3.3 HuMo - 音频驱动口型/表情生成

HuMo 支持从音频输入驱动视频中角色的口型和表情：
- 节点代码：`ComfyUI/custom_nodes/ComfyUI-WanVideoWrapper/HuMo/nodes.py`
- 示例工作流：`wanvideo_2_1_14B_HuMo_example_01.json`

适用场景：需要角色说话/唱歌的视频。

### 3.4 MultiTalk - 音频驱动多人对话

MultiTalk 支持从音频驱动多角色之间的对话唇形：
- 节点代码：`ComfyUI/custom_nodes/ComfyUI-WanVideoWrapper/multitalk/nodes.py`

### 3.5 LongCat Avatar - 音频+图片到说话视频

LongCat 提供端到端的音频驱动说话头像方案：
- 示例工作流：`LongCatAvatar_audio_image_to_video_example_01.json`

### 3.6 外部后期配音方案（不需要额外模型）

如果不想在 ComfyUI 中集成音频生成，可以采用后期处理方案：

1. 在 ComfyUI 中生成无音频的视频
2. 使用以下任一工具添加配乐：
   - **ElevenLabs**：AI 语音生成 + 音效
   - **Suno / Udio**：AI 音乐生成
   - **Adobe Premiere / DaVinci Resolve**：专业视频剪辑+配乐
   - **FFmpeg 命令行**：直接合并音频轨道
     ```bash
     ffmpeg -i video.mp4 -i audio.mp3 -c:v copy -c:a aac -shortest output.mp4
     ```

## 4. 根据输入图片自动生成提示词

### 4.1 WanVideoWrapper 内置方案：Qwen LLM + Prompt Extender

**这是目前最直接、已集成好的方案**，无需额外安装插件。相关文件：
- `ComfyUI/custom_nodes/ComfyUI-WanVideoWrapper/qwen/qwen.py`
- `ComfyUI/custom_nodes/ComfyUI-WanVideoWrapper/qwen/system_prompt.py`

#### 工作原理

`WanVideoTextEncodeCached` 节点内置了可选的 `extender_args` 输入，当提供 Qwen LLM 模型时，会自动调用 `WanVideoPromptExtender` 对你的提示词进行改写和扩展。

对于 **I2V（图生视频）** 场景，系统内置了以下专用模板：

| 模板名称 | 语言 | 用途 |
|----------|------|------|
| I2V Rewriter (Chinese) | 中文 | 根据图片内容和用户提示词，重写为强调动态动作的视频描述 |
| I2V Rewriter (English) | 英文 | 同上，输出英文 |
| I2V Imagination (Chinese) | 中文 | 不看用户提示词，仅根据图片内容**凭空想象**合理的动作描述 |
| I2V Imagination (English) | 英文 | 同上，输出英文 |

#### 使用步骤

1. **下载 Qwen LLM 模型**

   将 Qwen 模型文件放入 `ComfyUI/models/text_encoders/`（或 `ComfyUI/models/LLM/`）：

   - 推荐：`Qwen2.5-3B-Instruct`（约 6GB，显存友好）
   - 或：`Qwen2.5-7B-Instruct`（约 14GB，质量更高）

2. **节点配置方法 A：使用 WanVideoPromptExtenderSelect + WanVideoTextEncodeCached**

   ```
   QwenLoader ──> WanVideoPromptExtender ──> (手动模式)
   或
   WanVideoPromptExtenderSelect ──> extender_args ──> WanVideoTextEncodeCached (自动模式)
   ```

   - **WanVideoPromptExtenderSelect**：设置模型名称、系统提示词模板、最大 token 数，输出 `extender_args` 参数包
   - 将 `extender_args` 连接到 `WanVideoTextEncodeCached` 的 `extender_args` 输入
   - 在 `WanVideoTextEncodeCached` 的 `positive_prompt` 中，输入简短的关键词，LLM 会自动扩展

3. **系统提示词选择**

   对于图生视频（I2V），推荐选择：
   - 如果已有简单提示词：`I2V Rewriter (Chinese)` 或 `I2V Rewriter (English)`
   - 如果完全不想写提示词：`I2V Imagination (Chinese)` 或 `I2V Imagination (English)`——系统会**仅根据图片内容**自动写出完整的动态描述

4. **参数建议**

   | 参数 | 推荐值 | 说明 |
   |------|--------|------|
   | max_new_tokens | 256-512 | 生成提示词的最大长度 |
   | device | gpu | 使用 GPU 推理 |
   | force_offload | true | 生成后卸载模型以释放显存 |
   | system_prompt | I2V Rewriter (Chinese/English) 或 I2V Imagination | 选择模板 |

#### 完整自动化工作流示例

```
LoadImage ──────────────────────────────> WanAnimateToVideo
QwenLoader ──> WanVideoPromptExtenderSelect ──> WanVideoTextEncodeCached ──> WanAnimateToVideo
```

在这个工作流中：
1. `QwenLoader` 加载 Qwen LLM 模型
2. `WanVideoPromptExtenderSelect` 配置自动提示词扩展参数
3. `WanVideoTextEncodeCached` 接收原始图片（通过其内置的 clip vision 编码），结合 Qwen LLM 生成扩展后的提示词
4. 扩展后的提示词用于视频生成

**关键点**：`WanVideoTextEncodeCached` 节点本身就支持图片输入（通过 `clip_vision_output` 参数），当你启用了 extender 并使用 `I2V Imagination` 模板时，LLM 会基于图片编码 + 系统提示词，自动写出适合该图片的动态视频描述。

### 4.2 替代方案：外部 API 图片描述

如果不想在本地运行 LLM，可以使用外部 API：

1. **OpenAI GPT-4V / Claude Vision**：发送图片到 API，获取描述，再手动输入 ComfyUI
2. **HuggingFace 图片描述模型**：如 BLIP-2、LLaVA，可本地运行
3. **JoyCaption / WD14 Tagger**：从图片提取标签，适合动漫/二次元风格

### 4.3 推荐的自动化提示词策略

对于你的使用场景（固定角色 + 不同动作），建议：

1. **基础模式**：使用 `I2V Imagination (Chinese)` 模板 + Qwen LLM，完全自动生成
2. **引导模式**：在 positive_prompt 中写简短关键词（如"缓慢转头，微笑"），使用 `I2V Rewriter (Chinese)` 模板让 LLM 扩展为完整描述
3. **手动模式**：手动编写精细提示词，不启用 extender

## 5. 常见问题与调优建议

### 5.1 显存管理

| 配置 | 14B 模型预估显存 |
|------|-----------------|
| 512×768, 57帧, batch_size=1 | 约 14-16 GB |
| 增加帧数到 81 | 约 15-18 GB |

**显存不足时的优化方案**：
- 启用 block swap（在 WanVideoModelLoader 中设置），将部分 Transformer block 卸载到 CPU RAM
- 使用 `WanVideoSetBlockSwap` 节点精细控制
- 降低分辨率或帧数
- 使用 GGUF 量化版模型（WanVideoWrapper 已支持 GGUF 加载）
- 清理 Triton 缓存：删除 `C:\Users\Administrator\.triton` 和 `C:\Users\Administrator\AppData\Local\Temp\torchinductor_Administrator`

### 5.2 视频质量调优

| 调整项 | 效果 |
|--------|------|
| 提高 steps（8→12） | 更精细的画面细节，但时间翻倍 |
| 降低 CFG（1.4→1.0） | 更自然的动作，但可能偏离提示词 |
| 提高 CFG（1.4→2.0） | 更严格遵循提示词，但可能产生伪影 |
| 提高 DWPreprocessor resolution | 更精确的姿态估计 |
| 使用 Enhance-A-Video | 提升单帧画质（FETA 节点） |

### 5.3 模型文件位置总结

| 模型类型 | 存放目录 |
|----------|----------|
| Diffusion Model (UNET) | `ComfyUI/models/diffusion_models/` |
| Text Encoder (T5/CLIP) | `ComfyUI/models/text_encoders/` |
| VAE | `ComfyUI/models/vae/` |
| LoRA | `ComfyUI/models/loras/` |
| Clip Vision | `ComfyUI/models/clip_vision/` |
| Qwen LLM | `ComfyUI/models/text_encoders/` 或 `ComfyUI/models/LLM/` |
| MMAudio | `ComfyUI/models/mmaudio/` |

### 5.4 推荐模型下载源

- WanVideo 官方模型：https://huggingface.co/Kijai/WanVideo_comfy
- fp8 量化模型（推荐）：https://huggingface.co/Kijai/WanVideo_comfy_fp8_scaled
- Qwen LLM：https://huggingface.co/Qwen（推荐 Qwen2.5-3B-Instruct 或 7B-Instruct）

### 5.5 快速对照：你的需求 → 解决方案
