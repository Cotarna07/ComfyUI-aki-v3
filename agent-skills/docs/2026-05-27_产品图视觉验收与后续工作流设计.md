# 产品图视觉验收与后续工作流设计

> 日期：2026-05-27  
> 轨道：`factual_product`（真实商品展示）优先  
> 本机：ComfyUI `0.20.1`，`NVIDIA GeForce RTX 5070 Ti`，约 `16 GB VRAM`

## 核心结论

这轮问题不是简单的“提示词写得不够华丽”。在真实商品图任务里，生成模型
越自由，越容易用更好看的假商品替代真实商品。正确方向是把流程拆成两层：

1. 商品主体采用原图像素或经过审核的掩膜锁定，生成模型只负责背景、灯光和承托环境。
2. 生成后增加视觉验收门禁，并允许自动拒绝和重试；不能把任意一张漂亮图片直接标为商品图。

视觉分析模型确实应当加入流程，但它不是唯一裁判。本轮已经出现实证：本地
`Qwen3-VL` 对包含第二辆黑色汽车的 F1 合成成片给出通过；当问题被拆成
“仅检查未叠加商品的背景底板是否含有车辆”后，模型才正确报出车辆风险。

## 本轮实际观察

### 生成路线对照

| 路线 | 优点 | 本轮失败方式 | 结论 |
|------|------|--------------|------|
| `Flux.1 Kontext` 整图编辑 | 光影和背景统一，画面冲击力强 | 裁断赛车、删去警员/配件、将商品重画为近似物 | 仅可作创意候选，不能默认作真实商品图 |
| `RMBG-2.0` 前景锁定 + SDXL 背景 + 合成 | 商品结构、人物和附件可以保持原图像素 | 背景可能生成伪配件；角度不匹配会悬浮 | 真实展示的当前安全主线 |
| 锁定前景 + `staged` 展示台背景 | Creeper、红色跑车、魔法教室氛围更强 | F1 背景生成另一辆黑车；警车仍像悬挂在背景板前 | 必须在合成前审查背景底板 |

### 被拒绝的典型结果

| 商品 | 失败现象 | 风险类型 |
|------|----------|----------|
| Creeper 早期整图编辑 | 变成真实绿色越野车 | 商品身份漂移 |
| 红色跑车 Kontext 版 | 车体被画幅裁断 | 商品信息缺失 |
| 警车 Kontext 版 | 警员被删除、文字变形 | 配件缺失 / 虚构文字 |
| 魔法教室 Kontext 版 | 多个人仔和附件缺失 | 套装数量错误 |
| F1 `staged` 锁前景版 | 背景底板出现一辆黑色汽车 | 背景伪造商品 / 误导性重复对象 |

### 可保留的安全基线

当前新增工作流 [product_locked_background_sdxl.json](../comfyui/workflows/api/product_locked_background_sdxl.json)
采用：

- 前景：`RMBG-2.0` 分离后，将原始商品像素重新合成，不让扩散模型重画主体。
- 背景：`juggernautXL_v9Rdphoto2Lightning.safetensors`，仅生成摄影棚或空展示场景。
- 合成：以掩膜生成柔和接触阴影，再覆盖原商品前景。
- 留痕：保存 `background`、`foreground_audit` 与 `locked_factual` 三种输出。

这条路线解决了最危险的主体重绘问题，但尚未自动解决遮罩误删、背景伪物体和
接地关系不自然的问题，必须补上验收环节。

## 本机已有能力

| 能力 | 本机状态 | 适合职责 | 不应承担的职责 |
|------|----------|----------|----------------|
| `RMBG-2.0` | 已安装，约 844 MiB | 前景初始掩膜、原像素合成 | 判断附件是否齐全 |
| `ComfyUI-Florence2` | 节点已加载；模型权重需按需确认/安装 | 区域描述、开放词汇检测、OCR/文字污染定位 | 单独决定商品一致性 |
| `Grounding DINO` 权重 | 已存在 | 以“车/人仔/包装/文字”等词定位禁物 | 判断细微结构相同 |
| `SAM2` / SAM 权重 | 节点与若干权重已存在 | 对已定位物体做精细掩膜与边界检查 | 识别商品身份 |
| `comfyui-ollama` | 节点已加载 | 在图内调用 VLM 诊断 | 直接替代发布验收规则 |
| `Qwen3-VL` 8B Q4 | Ollama 已有非官方改版，约 6.1 GB | 本机诊断与工作流原型验证 | 正式发布唯一审批器 |

本轮实测还发现：ComfyUI 生成模型驻留时启动 Ollama VLM 会发生内存布局失败；
调用 ComfyUI `/free` 卸载模型后，VLM 才能正常审查。因此在 16 GB 本机上，
生成和视觉审查应当串行错峰执行。

## 推荐工作流

真实商品图不要设计为“一张 ComfyUI 大图里串到底”。标准节点图不适合负责
条件拒绝、换 seed 重试和审查报告归档。建议由 Python 编排器驱动数个 API 阶段：

```text
原始多角度图片
  -> 身份清单与选图
  -> 前景掩膜/附件覆盖审计
  -> 只生成背景底板
  -> 背景底板禁物验收 [失败则换 seed/收紧场景]
  -> 原商品像素 + 接触阴影合成
  -> 最终接地/遮挡/文字污染验收 [失败则退回对应阶段]
  -> 人工终验后标为 factual_product
```

### 阶段 0：身份清单与选图

输入同一 SKU 的多张原图，输出一份 `identity_manifest.json`：

```json
{
  "product_id": "1005009067934624",
  "track": "factual_product",
  "must_keep": [
    "one red brick-built Formula 1 car",
    "four tires",
    "front wing, cockpit and rear wing"
  ],
  "forbidden_generated_objects": [
    "another vehicle",
    "spare wheel",
    "minifigure not present in the selected source",
    "brand or specification text not copied from source"
  ]
}
```

选图先于美术生成：完整、落地视角、无包装文案遮挡的原图优先。素材本身呈
俯视悬浮视角时，背景模型无法可靠补出自然接地关系。

### 阶段 1：主体锁定

`factual_product` 默认不用整图重绘。对选定原图执行 `RMBG`，复杂套装可用
Grounding DINO/SAM2 补查小配件边界，并保存：

- `foreground_audit.png`：黑底/透明底前景审计图。
- `mask.png`：供后续边界与覆盖率复核。
- `identity_manifest.json`：该图必须保留的零件和人物。

人物、轮胎、武器、瓶子、透明灯条等细小关键件一旦不在前景审计图内，当前
输入直接失败，不能寄希望于背景生成模型补回来。

### 阶段 2：背景底板生成

背景提示词必须描述“空场景”，不描述商品本身。建议保留两种模式：

| 模式 | 用途 | 风险 |
|------|------|------|
| `studio` 空棚 | 真实性基线，适合主图或批量 SKU | 画面较保守 |
| `staged` 空展示台 | 广告氛围候选，允许台面/地面/灯具 | 可能生成与商品相似的道具 |

背景节点只输出 `background_plate.png`，尚未叠加商品。这样视觉门禁可以看到
所有由背景模型私自添加的对象，不会被前景商品遮住。

### 阶段 3：背景底板门禁

这是本轮新增的关键门禁，必须发生在合成之前。

1. Grounding DINO / Florence2 查询禁物类别，例如 `car`, `wheel`, `toy`,
   `minifigure`, `box`, `text`, `logo`。
2. Qwen3-VL 接收“仅背景底板”和具体禁物表，输出结构化判断。
3. 规则引擎取严格并集：任一检测器报告商品相似对象，或 VLM 的证据文本提到
   存在禁物，即使它的 `pass` 字段写错，也一律拒绝。

本轮 F1 的底板用聚焦问题审查后正确得到：

```json
{
  "has_vehicle_like_object": true,
  "objects": ["sports car"]
}
```

而无车辆的普通空棚底板得到：

```json
{
  "has_vehicle_like_object": false
}
```

背景门禁失败的重试策略是换 seed、回退到 `studio`、收紧背景对象列表，绝不
重新生成商品主体。

### 阶段 4：合成与接触阴影

只有通过背景门禁的底板才能进入合成：

- 原商品前景保持像素锁定。
- 接触阴影可以由前景掩膜模糊、下移并降低不透明度生成。
- 不使用能重新绘制商品内部的 relight 或 img2img 节点作为真实展示默认路径。

如果出现悬浮感，优先修正选图、台面高度、阴影方向和背景构图；不要放开主体
重绘来伪造落地效果。

### 阶段 5：最终门禁

最终成片必须独立检查以下项目：

| 检查层 | 检查内容 | 失败处理 |
|--------|----------|----------|
| 掩膜层 | 前景审计图是否保留身份清单中所有人物和配件 | 换原图或修掩膜 |
| 语义层 | 是否多车、多人物、多配件、伪文字或商品类背景道具 | 拒绝并重生成背景 |
| 接地层 | 是否悬浮、穿插、台面透视明显不符 | 换背景/选图/阴影设置 |
| 人工层 | 结构、颜色、透明件、贴纸与配件逐项核验 | 未通过不得发布 |

VLM 适合输出“哪里可疑”和“应回退哪个阶段”，不允许覆盖硬规则或人工终验。

## 视觉模型选型

### 本机实施顺序

| 优先级 | 模型/组件 | 用途 | 说明 |
|--------|-----------|------|------|
| 1 | `RMBG-2.0` + 原图合成 | 保住主体像素 | 已可运行 |
| 2 | Florence-2 large/large-ft | 背景禁物、文字污染、区域描述 | 节点已存在；体量适合先接入 |
| 3 | Grounding DINO + SAM2 | 检测证据与边界掩膜 | 对禁物与细件提供可视证据 |
| 4 | 官方 Qwen3-VL 8B Instruct | 结构化语义审查 | 应替换当前非官方改版作正式本地候选 |

### 云端质量扩展

本机验证门禁逻辑后，云端可以把 VLM 审查升级到官方更大参数版本，例如
Qwen3-VL 32B 或 MoE Instruct 版本；生成模型也可升级为同路线更强编辑模型。
云端的意义是提高识别和生成上限，不是取消像素锁定、禁物门禁或人工核验。

## 显存与编排约束

在当前 16 GB 设备上，推荐执行顺序：

```text
1. ComfyUI 生成 N 个 background_plate
2. POST /free 卸载扩散模型与释放显存
3. 启动/调用 Ollama VLM 审查 N 个 background_plate
4. 只将通过的底板提交合成
5. 再次 /free 后审查最终成片
```

不要让 SDXL/Flux 与 Qwen3-VL 同时驻留，也不要因显存问题改用会降低交付质量
的生成参数。算力不足时保留同一质量路线，转云端执行。
