# agent-projects 三项目能力选型总览

这份文档面向“遇到问题时先判断该看哪个项目”的场景，聚焦以下三个项目：

- manga-anime-pipeline
- comfyui-test-instance
- comfyui-test-harness

结论先说：

- 如果你的目标是“把漫画章节结构化，并批量送进 ComfyUI 继续生成”，优先看 manga-anime-pipeline。
- 如果你的目标是“隔离插件、端口、输出目录，避免把主 ComfyUI 环境搞乱”，优先看 comfyui-test-instance。
- 如果你的目标是“先检查工作流、节点、模型、参数、链接是否有问题，再决定要不要运行”，优先看 comfyui-test-harness。

## 一页速览

| 项目 | 核心定位 | 主要输入 | 主要输出 | 最适合解决的需求 |
| --- | --- | --- | --- | --- |
| manga-anime-pipeline | 私有化漫画转动漫流水线 | 章节 manifest、漫画页图片、各阶段配置 | window manifest、structured packets、shot manifest、角色圣经、门禁报告、ComfyUI 提交结果 | 漫画长图切片、OCR、对白抽取、检测、导演分镜、批量投递 ComfyUI |
| comfyui-test-instance | 第二套并行 ComfyUI 测试实例 | 主 ComfyUI 代码与模型、插件清单、启动参数 | 独立 user/input/output/temp/database、自定义插件启用集 | 风险插件测试、工作流冒烟、与主实例并行运行、隔离输出和数据库 |
| comfyui-test-harness | ComfyUI 资源自动化校验框架 | 工作流 JSON、模型目录、ComfyUI 服务状态 | 预检结果、工作流校验结果、模型检查结果、Markdown 报告 | 缺模型、缺节点、参数异常、链接错误、服务器未启动等问题的快速定位 |

## 1. manga-anime-pipeline

### 核心功能

这个项目不是单个脚本，而是一条分阶段流水线，目标是把“漫画章节输入”整理成“可继续生成的结构化任务”。当前主线已经覆盖：

1. 读取章节 manifest 和页面图片。
2. 对长图做窗口切片。
3. 执行 OCR，并可选基于 OCR 结果整理对白与拟声词。
4. 执行 lightweight detection 或接入更强检测入口。
5. 生成 director 草稿和 shot manifest。
6. 生成 character bible，作为后续角色一致性的锚点。
7. 把 shot manifest 批量提交到 ComfyUI API，并生成门禁和验收报告。

项目里已经提供多层门禁脚本，适合逐段排查：

- Stage 3A：OCR 与 OCR-based Dialogue 环境和结果检查。
- Stage 4A：检测阶段门禁。
- Stage 5：Qwen3-VL director 门禁。
- Stage 6：ComfyUI 批量提交门禁。
- run_all_gates.py：按顺序串联所有门禁并输出总报告。

### 它能解决什么需求

它最适合下面这类需求：

- 我有漫画章节图片，想自动切片、识别文字、组织对白和分镜信息。
- 我想把前处理和 ComfyUI 投递串起来，而不是手动一张张整理。
- 我想为后续角色一致性、镜头规划、批量生成保留结构化中间产物。
- 我想在进入正式生成前，用门禁报告判断是哪一阶段卡住。
- 我想把一批 shot 按 workflow_route 自动映射到对应 ComfyUI 模板并批量提交。

如果你的问题本质上是“如何把漫画输入转成结构化镜头任务，并与 ComfyUI 生产链连接”，这个项目就是主战场。

### 这个项目的优势

- 有明确的阶段边界，适合按阶段排障，而不是全部混在一起。
- 中间产物完整，包含 windows、packets、shot manifest、character bible、acceptance report。
- 对 ComfyUI 的接入是 API 化的，可以做批量、复跑、追踪和验收。
- 支持 provider 化扩展，OCR、对白、检测、director 都不是写死的一条路。

### 当前边界和不适合的问题

当前还不适合把它理解成“完整成片系统”。README 已明确列出一些未接入或未完成的部分：

- Manga Image Translator 还未接入。
- 完整口型、音频、BGM 后期链未接入。
- 人工审核台未接入。

因此，如果你的问题是“我要直接做完整后期成片，含配音、口型、音乐、人工审核闭环”，这个项目目前只能覆盖前半段和 ComfyUI 提交，不是全流程最终解。

## 2. comfyui-test-instance

### 核心功能

这个项目的目标很明确：在当前工作区再起一套与主实例并行的 ComfyUI 测试实例，但尽量复用已有资源，避免重复复制整套 ComfyUI。

它默认采用“中隔离”方案：

- 共享主 ComfyUI 代码。
- 共享主 models 目录。
- custom_nodes 按 config/plugins.json 指定清单建立 Junction。
- user、input、output、temp 独立。
- 数据库独立。
- 可以选择再创建自己的 overlay .venv。

默认监听 127.0.0.1:8190，可以和主实例 8188 并行在线。

### 它能解决什么需求

这个项目最适合下面这类需求：

- 我想测试一批新插件，但不想直接污染主实例。
- 我想验证某个工作流是否能跑通，又不想把主实例的 output、temp、user 数据搅乱。
- 我想同时保留主实例生产环境和测试环境。
- 我想为某些插件单独装 requirements，而不是把主 Python 环境越装越乱。
- 我想做 MMAudio、VFI、Florence2、VideoHelperSuite 等节点组合测试，但需要一个可随时清理和重建的沙盒。

如果你的问题本质上是“需要一个安全试验场来跑 ComfyUI 工作流或插件”，这个项目就是优先入口。

### 这个项目的优势

- 初始化和启动逻辑清晰，setup_instance.ps1 与 start_instance.ps1 已经把目录、Junction、数据库隔离处理好了。
- 插件依赖可单独安装到 overlay .venv，不强制污染主实例运行时。
- 与主实例能并行运行，适合做回归对照和风险验证。
- 已经针对 Windows PowerShell 5.1 做过兼容性收敛，例如脚本提示文本改为 ASCII，避免中文解析问题。

### 当前边界和不适合的问题

它不是完整的环境复制，也不是容器化隔离。

- 主代码仍然共享，所以如果你的目标是“完全脱离主 ComfyUI 代码做隔离”，它还不够强。
- 模型目录也共享，所以它解决的是实例级隔离，不是模型资源级隔离。
- 它本身不负责判断工作流有没有配置错误，只负责提供一个更安全的测试实例。

因此，如果你的问题是“为什么这个工作流 JSON 结构有问题、缺哪个节点、缺哪个模型”，仅靠这个项目还不够，通常要配合 comfyui-test-harness。

## 3. comfyui-test-harness

### 核心功能

这个项目是自动化测试框架，重点不在“运行生产任务”，而在“先判断资源是否具备运行条件”。它当前覆盖三类核心检查：

1. 预检
   检查 ComfyUI 服务器是否可达、GPU/显存信息、已注册节点清单、models 全景。
2. 工作流校验
   校验工作流类型、节点类型、模型引用、参数范围、链接完整性。
3. 模型检查
   检查重点模型是否在磁盘上，尤其是近期新增的 checkpoint、LoRA、MMAudio 模型、RIFE VFI 依赖。

它支持两种模式：

- focus：默认只看近期重点工作流和近期新增资源。
- all：全库巡检，适合做大范围盘点。

它也支持离线模式。ComfyUI 没启动时，仍然可以做静态校验，只是拿不到在线节点清单。

### 它能解决什么需求

这个项目最适合下面这类需求：

- 我不知道工作流跑不起来，到底是节点没装、模型没放对、还是参数写得离谱。
- 我想在真正开跑前，先做一次自动化巡检。
- 我只想重点检查最近新增的 MMAudio / API 工作流，不想被大量历史工作流噪音淹没。
- 我需要一份 Markdown 测试报告，方便回看和对比。
- 我希望在 ComfyUI 未启动时，先离线发现静态错误。

如果你的问题本质上是“先做资源健康检查和配置排错”，这个项目应该先于正式运行使用。

### 这个项目的优势

- 很适合定位缺失类问题，比如缺模型、模型放错目录、节点类型未注册、链接断裂。
- 默认 focus 模式已经把重点放在近期 MMAudio 和 API 工作流上，适合当前工作区的实际关注面。
- 通过环境变量 COMFYUI_SERVER_URL，可以把检查目标切到其他 ComfyUI 实例，而不局限于主实例。
- 报告输出标准化，适合作为排障起点，而不是靠肉眼翻 JSON。

### 当前边界和不适合的问题

它更像“资源与配置体检”，不是“最终效果评测器”。

- 它能告诉你工作流结构和依赖有没有明显问题，但不能替代人工判断画面美术质量。
- 它能发现参数超出常规范围，但不会自动替你完成创作级调参。
- focus 模式偏向近期重点资源，不代表全库其他历史工作流完全没问题。

因此，如果你的问题是“画面风格为什么不够好、角色为什么不够稳、镜头叙事是否成立”，它只能帮你排除基础配置问题，不能单独给出创作结论。

## 怎么选：按问题类型找项目

### 先看这个映射

- 漫画章节如何切片、OCR、抽对白、生成镜头任务：看 manga-anime-pipeline。
- 新插件或高风险工作流怎么单独试，不影响主实例：看 comfyui-test-instance。
- 工作流为什么打不开、节点为什么报缺失、模型是不是没放对：先看 comfyui-test-harness。
- 想批量把镜头任务投递给 ComfyUI：看 manga-anime-pipeline。
- 想给测试实例单独安装插件依赖：看 comfyui-test-instance。
- 想先离线扫一遍重点工作流和模型：看 comfyui-test-harness。

### 推荐组合

很多实际问题，最有效的方式不是单看一个项目，而是组合使用：

1. 新工作流 / 新插件接入
   先用 comfyui-test-instance 起隔离环境，再用 comfyui-test-harness 检查节点、模型和工作流结构。

2. 漫画到分镜到批量提交
   用 manga-anime-pipeline 负责章节解析、结构化和提交；如果不想打扰主实例，可以把它的 ComfyUI 目标配置切到第二实例。

3. 排查“到底是环境问题还是流程问题”
   先用 comfyui-test-harness 排除缺模型、缺节点、参数和链接问题；如果怀疑插件冲突，再转到 comfyui-test-instance 做隔离复现；确认环境没问题后，再回到 manga-anime-pipeline 处理业务流程。

## 如果你现在要找解决方案，建议这样入手

### 你的问题偏环境或依赖

优先顺序：

1. comfyui-test-harness
2. comfyui-test-instance

适用关键词：

- 节点没装
- 模型缺失
- 模型目录不对
- 工作流 JSON 有问题
- 参数异常
- 插件依赖冲突

### 你的问题偏业务流程或批量生产

优先顺序：

1. manga-anime-pipeline
2. comfyui-test-harness

适用关键词：

- 漫画长图处理
- OCR 与对白抽取
- 结构化分镜
- 角色一致性中间产物
- 批量提交 ComfyUI
- 门禁报告与阶段排障

### 你的问题偏“安全试验场”

优先顺序：

1. comfyui-test-instance
2. comfyui-test-harness

适用关键词：

- 并行起第二实例
- 独立 output/temp/user
- 单独测试插件组合
- 不污染主实例

## 最后的选型判断

可以把这三个项目理解成三个层次：

- comfyui-test-harness 负责“先查清楚能不能跑”。
- comfyui-test-instance 负责“给你一个更安全的地方去跑”。
- manga-anime-pipeline 负责“把业务输入整理成可批量执行的生成任务，并推进到 ComfyUI”。

如果你后面是带着具体问题来查方案，通常可以先按下面一句判断：

- 先诊断资源和工作流问题：看 comfyui-test-harness。
- 先隔离环境和插件风险：看 comfyui-test-instance。
- 先推进漫画到镜头任务和批量生成：看 manga-anime-pipeline。
