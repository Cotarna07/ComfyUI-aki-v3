# 漫画分镜上下拆分问题修复方案

## 问题确认

**📌 重要更新（2026-05-09）**：以下"高度阈值 + 像素边界"方案已被更优的"内容感知分镜检测"方案取代。详见 `2026-05-09_漫画分镜智能检测与合图方案.md`。本文档保留作为问题描述和早期简单方案的记录。

核心思路已从"补丁式拼接"升级为"先理解分镜构图，再决定切割点"。推荐首先使用 DeepPanel（GitHub: pedrovgs/DeepPanel）或 YOLOv12x Comic Panel Detector（HuggingFace: mosesb/best-comic-panel-detection）做内容级分镜框检测，然后基于检测结果判断是否需要合图。

以下为原始分析内容（已归档）：

通过对 `D:\openclaw_tools\downloads\picaweb_manga` 中漫画图片的实际尺寸检测，确认以下拆分模式：

### 条漫类（美丽新世界、继母的朋友们、乖乖上钩）

| 特征 | 数值 |
| --- | --- |
| 图片等宽 | 全部 720px |
| 最大分块高度 | 4200px |
| 实际高度范围 | 661px ~ 4200px |
| 拆分原因 | picaweb 服务端按约 4200px 纵向切块下发，切割点不保证对齐分镜边界 |

**实际案例（继母的朋友们 ep001）**：

```
p006: 720x1983  ← 一张短图，可能恰好落在分镜末尾
p007: 720x4200
p008: 720x661   ← 极短残片，几乎肯定是上一张图的延续
p009: 720x4200
p010: 720x1349  ← 残片
p011: 720x4200
p012: 720x1272  ← 残片
```

`p008` 高仅 661px，它不是独立分镜——它是上一张 `p007`（4200px）的底部延续，一个完整分镜被切割点硬拆成了上下两半。

### 页漫类（蚀むハイヴ、社團學姊）

蚀むハイヴ的图片高度各异，属于标准页漫，没有按固定高度切块的问题。页漫的拆分问题通常表现为左右跨页被分成两张——当前这批数据中未发现这一问题，暂不作为优先修复目标。

## 核心修复目标

**条漫纵向拆分**是当前最需要解决的问题：保证每个完整分镜保留在单张图片内，不被纵向切块边界切断。

## 修复方案

### 方案 A：像素级边界对比 + 自动合图（推荐）

**原理**：条漫的 `720x4200` 切块边界如果正好落在分镜中间，那么切线上下的像素是连续的（同一画面内容被一刀两断）。通过比较相邻两张图的"上张底部"与"下张顶部"的像素，判断是否属于同一分镜。

**具体步骤**：

1. **提取边界条带**：
   - 从图片 N 底部取 8px 高的水平条带
   - 从图片 N+1 顶部取 8px 高的水平条带

2. **计算相似度**：
   - 两张条带逐像素比较，计算 MSE 或 SSIM
   - 如果相似度超过阈值，判定为"同一分镜被拆分"

3. **合图处理**：
   - 将判定为连续的图片垂直拼接（stitch）
   - 输出拼接后的完整分镜图片

4. **迭代**：
   - 拼接后如果仍然与下一张图片连续，继续拼接
   - 直到遇到独立分镜边界或长度达到合理上限

**优点**：检测准确、自动修复、不依赖任何外部模型
**缺点**：对跨页留白、纯黑过渡等情况的边界判断可能失效

### 方案 B：基于分镜框线的检测（辅助方案）

**原理**：漫画分镜通常由框线包围。如果一张图的底部没有闭合的分镜框线，说明它被从中间切断。

**具体步骤**：

1. 对每张图片做边缘检测或 line detection
2. 检查图片底部 20% 区域是否存在水平分镜框线
3. 若无框线 + 底部像素不完全是背景留白 → 标记为"疑似被切断"
4. 结合方案 A 的像素对比做交叉验证

**优点**：适合标准页漫（有明显分镜框线的漫画）
**缺点**：条漫通常没有明显分镜框线，适用性有限

### 方案 C：高度阈值 + 宽松合图（兜底方案）

**原理**：条漫中高度显著低于 4200px 的图片几乎不可能是独立分镜（一个分镜通常不会只有 661px 高）。

**具体步骤**：

1. 统计全书图片的高度分布
2. 设定阈值：高度低于最大高度 30%（如 < 1200px）的图片，自动与上一张拼接
3. 对阈值以上的图片，用方案 A 做精确判断

**优点**：简单粗暴、无漏网之鱼
**缺点**：可能误将"确实很短的独立分镜"也拼接掉

### 推荐组合策略

三者互补：

1. **方案 C 做粗筛**：高度 < 最大块高度 × 0.3 的图片，直接标记为"需要拼接"
2. **方案 A 做精判**：对粗筛之外的相邻图片做像素级边界对比
3. **方案 B 做页漫适配**：只在页漫类型中启用，辅助判断非标准分镜边界
4. **最终动作**：将所有判定为"同一分镜被拆分"的图片垂直拼接

## 处理流程图

```
输入: 按 CSV 顺序排列的图片列表
      ↓
判断漫画类型（条漫 / 页漫）
      ↓
条漫路径                        页漫路径
      ↓                              ↓
方案C粗筛: 高度<阈值?           方案B: 检查底部框线
      ↓ Yes                          ↓ No框线
直接标记需拼接                 ↓         ↓
      ↓                      标记需拼接  方案A像素对比
方案A精判: 其余相邻图对比               ↓
      ↓                           确认是否拼接
标记 "same_panel" 布尔值              ↓
      ↓                         输出拼接分组
按分组执行垂直拼接
      ↓
输出修复后的图片 + 映射表
```

## 输出格式建议

修复后应输出一份映射表（JSON），记录原始图片与修复后图片的关系：

```json
{
  "comic": "继母的朋友们",
  "episode": 1,
  "original_count": 14,
  "fixed_count": 10,
  "stitch_groups": [
    {
      "group_id": 1,
      "source_files": ["ep001_p007.jpg", "ep001_p008.jpg"],
      "output_file": "fixed/ep001_p007_stitched.jpg",
      "stitch_type": "vertical",
      "reason": "bottom_pixels_match + height_661_below_threshold"
    }
  ]
}
```

## 实验验证建议

1. **优先验证对象**：继母的朋友们 ep001（数据典型，高度变化大）
2. **对比基线**：先跑方案 C（纯高度阈值），再看方案 A 能额外发现多少漏网之鱼
3. **人工抽查**：随机抽查 10 组拼接结果，确认没有误拼接

## 对后续流程的影响

这个修复是整个漫画理解链路的**前置预处理**。修复完成后：

1. 分镜切割的输入变为"已修正的完整分镜图片"，不再有残片
2. OCR 文字识别不再因分镜断裂而丢失对白上下文
3. 故事连续性理解不再因缺页而产生剧情断层
4. 动画提示词生成不再基于残缺画面
