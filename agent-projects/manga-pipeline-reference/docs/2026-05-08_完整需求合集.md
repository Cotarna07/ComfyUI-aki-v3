# 漫画分镜项目完整需求合集

## 这份文档是干什么的

这份文档不是阶段性笔记，而是给后续新对话模型使用的“总需求交接底稿”。目标是把从本次任务开始到现在已经明确过的业务目标、功能要求、设计逻辑、工程约束、参考资源和已完成基础设施一次性说清，避免新对话里重复做需求澄清和资源检索。

结论先说：最终要实现的不是单个 ComfyUI 工作流，而是一条围绕 ComfyUI 组织的本地漫画理解与角色连续性处理链路。它至少要解决四件事：自动切分分镜、按故事连续性排序、生成带上下文的分镜故事描述、识别跨分镜同一角色，并为后续连续故事生成或角色一致性生成提供基础。

## 一、最终总目标

要落地的整体目标是：

1. 输入整页漫画图片，支持黑白漫画和彩色漫画。
2. 自动切分为独立分镜，并保留每个分镜的几何信息。
3. 不只给出几何阅读顺序，还要给出更符合剧情推进的故事连续性顺序。
4. 对每个分镜做内容理解，输出带上下文关系的故事描述，而不是孤立 caption。
5. 在多个分镜之间识别同一角色，建立稳定的角色编号与角色注册表。
6. 为后续连续故事生成、角色一致性生成或素材检索提供结构化结果。

## 二、从最开始到现在确认过的全部硬性需求

### 1. 漫画分镜切割与排序

必须满足：

1. 基于 ComfyUI 搭建工作流或以 ComfyUI 为核心能力源。
2. 支持黑白漫画和彩色漫画的自动分镜切割。
3. 切割结果必须能导出为独立分镜图片，而不只是视觉预览。
4. 切割后必须保留 `crop`、`polygon`、`bbox`、候选阅读顺序等中间信息。
5. 排序目标不是单纯“视觉阅读顺序”，而是“故事连续性顺序”。
6. 必须优先查找、匹配、复用现成可用的同类开源工作流模板或节点资源，而不是完全闭门自造。

### 2. 分镜内容识别与上下文故事生成

必须满足：

1. 对切割后的单张分镜做内容理解。
2. 输出结果不能只有一句泛化 caption，必须包含剧情上下文关联。
3. 结果要能描述角色、动作、场景、对话文本、情绪、前后衔接线索等信息。
4. 多个分镜之间要能形成连贯的故事描述，而不是每张图各说各话。

### 3. 跨图角色一致性识别

必须满足：

1. 能识别不同分镜中是否为同一人物角色。
2. 角色识别不能只靠单次视觉猜测，要有稳定的角色编号或角色注册表。
3. 结果必须服务于“连续故事生成时避免变脸、保持人物设定统一”这一目标。
4. 必须区分“识别任务”和“后续生成时的身份保持任务”，不能把两者混为一谈。

### 4. 全链路实现咨询与落地

必须满足：

1. 需要模型协助梳理从切割、排序、角色识别到故事生成的完整技术实现方案。
2. 需要把每一步对应的现成开源项目、ComfyUI 节点、示例工作流和工具资源整理出来。
3. 不是只要理论分析，后续是要在当前工作区里继续落地实现。

## 四、已经确认的设计逻辑和技术路线

### 1. 整体架构不是“一条龙巨型工作流”

已经确认的设计逻辑是：

1. 不要继续寻找一个幻想中的“一键全自动全链路 ComfyUI 工作流”。
2. 更稳妥的路线是模块化：分镜切分 + 快速视觉理解 + 强推理重排 + 角色注册表。
3. 需要状态管理、缓存、角色归并、章节级上下文的部分，不适合全部硬塞在单个 ComfyUI 图里。

### 2. 推荐总链路

已经确认的总链路为 7 个阶段：

1. 整页漫画预处理
2. 分镜边界检测与切割
3. 生成每个分镜的 `crop`、`polygon`、`bbox`、候选阅读顺序
4. 对单个分镜做首轮视觉理解和文本抽取
5. 基于上下文做二轮故事连续性重排与故事摘要
6. 建立角色注册表并执行跨分镜角色归并
7. 导出结构化结果，供后续生成或检索流程使用

### 3. 输出必须结构化

已经确认：

1. 不能只输出图片结果。
2. 至少要保留一份 JSON 或 JSONL。
3. 后续应支持缓存和续跑，避免原图未变化时反复重算。

### 4. 分镜排序的核心判断

已经确认：

1. 几何阅读顺序只是一轮候选，不是最终答案。
2. 真正的故事连续性排序必须依赖 VLM 或 LLM 的二轮推理。
3. 需要输出 `candidate_orders`、`best_story_order` 和 `reason`，而不是只返回一个顺序。

### 5. 角色一致性的核心判断

已经确认：

1. “识别是不是同一角色”和“后续生成时保持同脸”是两件不同的事。
2. FaceAnalysis 更偏识别与比对。
3. InstantID / IPAdapter FaceID 更偏生成时的身份保持。
4. 真正可落地的角色归并要依赖三类信息：
   - 人脸相似度
   - 视觉属性相似度
   - 叙事线索相似度

### 6. 推荐实施形态

已经确认：

1. 验证版可以先用纯 ComfyUI 节点编排。
2. 正式版更推荐“ComfyUI 负责视觉节点，外部脚本负责编排”。
3. 外部编排需要负责排序、角色注册表、缓存、导出、续跑。

## 七、工作区内必须遵守的工程边界

这些不是业务功能，但新对话模型实现时必须遵守：

1. 默认只允许修改 `agent-projects/`、`agent-skills/` 以及固定入口说明文件。
2. `ComfyUI/`、启动器原区、模型目录默认只读，除非用户明确授权。
3. 人类可读文档默认使用简体中文。
4. 命令示例默认使用 PowerShell，优先复用现有 `.venv`。
5. 不要访问或依赖任何名为 `cache` 或其变体的目录内容。
6. 模型文件、大体积素材、权重不要塞进 `agent-projects/` 或 `agent-skills/`。

## 八、性能与实现约束

后续正式实现时，应默认遵守以下约束：

1. 项目是 GPU-first，不要在应走 CUDA 的重路径上静默回退到 CPU。
2. 所有重推理链路应支持 batch 配置。
3. 长任务应支持分段、缓存和可重入。
4. 避免对未变化的输入重复计算 embedding、抽帧、切分结果或识别结果。
5. CPU 负责调度、状态管理、导出和文件组织；GPU 负责推理。
6. 关键阶段应记录耗时、缓存命中、异常回退与显存占用。

已经确认：

1. 参考资源项目 `manga-pipeline-reference` 主要用于留存第三方资源、文档和维护脚本。

4. 新项目应复用参考资源项目中的仓库、文档、manifest 和刷新脚本，而不是重新检索。
