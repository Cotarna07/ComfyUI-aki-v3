# 2026-06-01 批量商品图优化项目设计

本文档沉淀一个后续可开发的项目方案：批量读取指定目录下的商品图片，自动分析商品身份，生成每个商品的专属优化提示词，再调用本地 ComfyUI 工作流批量产出优化图，并生成可比较、可复跑、可验收的报告。

项目宿主为：

```text
agent-projects/product-media
```

## 项目目标

建立一套可插拔的商品图优化流水线：

1. 批量扫描用户指定目录中的商品图片。
2. 使用视觉分析模型提取商品事实：品类、颜色、结构、配件、场景归属、不可改元素。
3. 使用 LLM 根据商品事实和目标风格生成专属提示词。
4. 支持测试不同 VLM、LLM、ComfyUI 工作流、主模型、LoRA、步数、seed 和提示词规则。
5. 调用本地 ComfyUI API 自动提交任务、收集输出、记录参数。
6. 生成 HTML 报告，对比输入图、提示词、模型组合、输出图和风险结论。

最终目标不是一次性脚本，而是一个长期可扩展的批处理项目。

## 关键原则

商品图必须分两条轨道，不混用结论：

| 轨道 | 目标 | 默认策略 | 验收要求 |
|---|---|---|---|
| `factual_product` | 真实商品展示 | 优先锁商品主体、换背景、补光影 | SKU 结构、颜色、配件、人仔数量、贴纸/印花不能变 |
| `creative_campaign` | 创意广告视觉 | 可整体图生图、强化场景与动作 | 必须标注为创意图，不用于核验实物 |
| `diagnostic_preview` | 模型/提示词测试 | 快速低成本批测 | 只用于筛选，不交付 |

已验证经验：

- 场景语义必须对位：警车在城市道路，赛车在赛道，苦力怕在方块生存游戏场景，建筑套装用完整高视角或展示空间。
- 整图图生图容易重画主体，适合创意广告图，不适合真实商品展示图。
- 当前 Flux2 Klein Base 4B 低步数容易糊，提高步数能改善清晰度，但不能根治 SKU 漂移。
- 若要真实展示，必须优先走“商品前景锁定/合成/局部编辑”路线。

## 一键入口设计

建议后续主入口：

```powershell
D:\ComfyUI-aki-v3\.venv\Scripts\python.exe `
  D:\ComfyUI-aki-v3\agent-projects\product-media\scripts\batch_optimize_product_images.py `
  --input-dir "D:\path\to\product_images" `
  --batch-id "aliexpress_test_20260601" `
  --track creative_campaign `
  --provider-config "agent-projects/product-media/configs/providers.local.json" `
  --workflow-config "agent-projects/product-media/configs/workflows.flux2_klein.json" `
  --prompt-rules "agent-projects/product-media/configs/prompt_rules.product_scene.json"
```

常用参数：

| 参数 | 说明 |
|---|---|
| `--input-dir` | 用户指定的商品图目录 |
| `--batch-id` | 本次批次 ID，不传则按时间生成 |
| `--track` | `factual_product` / `creative_campaign` / `diagnostic_preview` |
| `--max-images` | 限制测试数量 |
| `--provider-config` | VLM/LLM 配置 |
| `--workflow-config` | ComfyUI 工作流矩阵 |
| `--prompt-rules` | 提示词规则 |
| `--dry-run` | 只生成计划和 API job，不提交 ComfyUI |
| `--no-wait` | 提交后不轮询，适合大批量排队 |
| `--resume` | 复用已有 facts/prompts，跳过已完成任务 |

    "yellow side stripe",

  "must_preserve": [
    "toy brick material",
    "four wheels",

    "extra vehicles covering product",

  "notes": "Only one view is available; exact hidden rear structure cannot be verified."

由 LLM 或本地规则生成，应缓存到 `prompts/<product_id>.json`。

    "preserve blue toy brick body",

## VLM 与 LLM 接口预留

### VLMProvider

用于商品事实提取与输出验收。

```python
class VLMProvider:
    def analyze_product(self, image_path: str, instruction: str, schema: dict) -> dict:
        ...

    def review_output(self, source_image: str, output_image: str, facts: dict, schema: dict) -> dict:
        ...
```

建议先支持 OpenAI-compatible 协议，方便接入本地或不同厂家：

```json
{
  "vlm": {
    "provider": "openai_compatible",
    "base_url": "http://127.0.0.1:1234/v1",
    "api_key_env": "LOCAL_VLM_API_KEY",
    "model": "local-vlm-model-name",
    "timeout_seconds": 120,
    "response_format": "json_schema"
  }
}
```

后续可增加适配器：

- `lmstudio_openai_compatible`
- `ollama`
- `openai_compatible_vendor`
- `custom_http_json`
- `mock_from_cache`

### LLMProvider

用于提示词生成、提示词改写和失败重试建议。

```python
class LLMProvider:
    def generate_prompt_plan(self, facts: dict, rules: dict, model_profile: dict, schema: dict) -> dict:
        ...

    def revise_prompt_after_review(self, prompt_plan: dict, review: dict, schema: dict) -> dict:
        ...
```

配置示例：

```json
{
  "llm": {
    "provider": "openai_compatible",
    "base_url": "http://127.0.0.1:1234/v1",
    "api_key_env": "LOCAL_LLM_API_KEY",
    "model": "local-llm-model-name",
    "temperature": 0.4,
    "timeout_seconds": 120
  }
}
```

要求：

- 所有 VLM/LLM 输出都必须 JSON 化，并做 schema 校验。
- 失败时保存原始响应到 `records/`，不能静默吞掉。
- 支持 `mock_from_cache`，方便不调用模型时复跑 ComfyUI。

## 提示词规则库

提示词应按产品类型和轨道生成，不要只套一个“奇幻卡通”模板。

### 通用结构

1. 商品事实锁定：保留类别、颜色、结构、数量、关键配件。
2. 场景归属：商品应在它自然属于的场景中。
3. 动作或镜头：车辆可飞驰，建筑套装可高视角长镜头，人仔套装可桌面舞台。
4. 画面质量：清晰主体、接触阴影、场景深度、材质、光影。
5. 禁止项：不加文字、logo、水印、包装、无关主体，不改 SKU。

### 例子

警车：

```text
Use the input blue brick police vehicle set as the locked hero product reference...
Show that same police car speeding along a miniature city road during an urgent patrol...
motion blur only in the background road lights...
```

赛车：

```text
Use the input red open-wheel brick race car as the locked hero product reference...
Show that same race car speeding on a grand-prix circuit, not parked...
```

建筑/场景套装：

```text
Test a wider long-shot composition from a high three-quarter overhead camera...
Keep the entire product sharp, centered, uncropped and readable...
```

## 批处理流程

```text
scan input dir
  -> copy inputs to runtime/<batch>/inputs_snapshot
  -> compute image hash and product_id
  -> VLM analyze product facts
  -> LLM generate prompt plan
  -> choose workflow variants
  -> convert/patch ComfyUI API graph
  -> submit ComfyUI jobs
  -> collect outputs
  -> VLM review factual/product drift
  -> generate contact sheet and HTML report
```

## 报告要求

HTML 报告至少包含：

- 批次信息、输入目录、执行时间、ComfyUI 版本、GPU。
- 每张输入图和商品事实摘要。
- 每个商品的提示词、轨道、风险标签。
- 工作流、模型、LoRA、steps、seed、耗时。
- 输出图网格。
- 跳过项及原因。
- VLM 验收：结构一致性、美术质量、文字/水印风险、是否可用于真实商品展示。
- 最终建议：最佳模型组合、失败原因、下一轮参数。

## 验收策略

### factual_product

任何关键结构漂移都应失败：

- 轮胎数变化。
- 车身类型变化。
- 人仔数量变化。
- 建筑套装结构变化。
- 颜色、贴纸、透明件、武器/配件变化。

推荐优先路线：

- 抠出商品原始前景。
- 生成背景。
- 合成接触阴影和环境光。
- 必要时只做轻微局部融合。

### creative_campaign

允许更强的光影、镜头、场景动作，但报告必须标注：

```text
创意广告图，不用于核验实物内容。
```

## MVP 开发阶段

### P0：本地可跑批处理

- 输入目录扫描。
- 手写/规则提示词。
- ComfyUI 工作流矩阵配置化。
- 自动生成 HTML 报告。
- 不依赖 VLM/LLM API。

### P1：接入 VLM/LLM

- OpenAI-compatible VLM/LLM 适配器。
- JSON schema 输出校验。
- facts/prompts 缓存。
- 支持 `mock_from_cache`。

### P2：质量验收闭环

- VLM 对比输入/输出。
- 自动标记 SKU 漂移。
- 支持失败原因汇总和 prompt 修订建议。

### P3：多模型/多 LoRA 实验平台

- 多 workflow、多主模型、多 LoRA、多 steps、多 seed 矩阵。
- HTML 报告支持横向比较。
- 输出最佳组合建议。

### P4：轻量 UI 或 Web 面板

- 选择输入目录。
- 选择轨道和模型配置。
- 查看队列、结果和报告。

## 风险与边界

1. 当前整图图生图工作流无法保证真实 SKU 完全一致。
2. 提示词能降低漂移，但不能替代前景锁定、遮罩、ControlNet 或局部编辑。
3. 不同模型对同一提示词理解差异很大，提示词需要按模型族适配。
4. VLM 只能提供风险证据，不能单独放行真实商品展示图。
5. 本地 16GB 显存适合做实用路线验证；更大模型和高分辨率最终制作可迁移到云 GPU。
